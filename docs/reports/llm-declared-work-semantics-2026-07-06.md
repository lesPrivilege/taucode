# LLM-Declared Work Semantics

2026-07-06. This note answers whether the current V3-WS anchor is structured
extraction, whether the model should return a JSON work-semantics block itself,
how that relates to view-based context, what DeepSeek reasoning replay implies,
and how this compares with frontier/server-side compaction.

## Short Answer

The current `TAUCODE_SEMANTIC_ANCHOR` is **structured deterministic extraction**.
It is generated by the harness from tool events: reads, edits, failed edits,
tests, and pending acceptance targets. The model does not author it.

A model-authored JSON work-semantics block is a valid next design, but it should
be treated as an **intent declaration**, not as an authority. The harness can use
it to decide which views are eligible for compaction only after checking the
claim against mechanical evidence: path, hash, turn, edit status, and future
re-read behavior.

So the view-based-context design is now one mechanism with three faces:

```text
model -> harness: intent declaration
harness storage: semantic ledger
harness -> model: trust/provenance hints
```

All three should share the same `path#hash` addressing. That makes it one
implementation, not three separate systems.

## Current Anchor vs. Proposed JSON Block

### Current Anchor

Source of truth: harness events.

Example facts:

- `read: path@hash`
- `edits: path +N -M hash old->new`
- `tests: command -> result`
- `pending: artifact.md`

Properties:

- deterministic;
- zero extra model output;
- cannot hallucinate a file hash or edit result;
- weak at intent and semantic completeness, because it only knows what happened,
  not what the model believes is now safe to forget.

### Proposed JSON Work-Semantics Block

Source of truth: model self-report.

Sketch:

```json
{
  "work_semantics": {
    "views": [
      {
        "path": "packages/core/src/compaction.ts",
        "hash": "abc12345",
        "retention": "semantic",
        "semantic_complete": true,
        "may_compact_raw": true,
        "summary": "Projection logic was split into compaction/projection.ts; public API remains re-exported."
      }
    ],
    "pending": ["SUBSYSTEM-MAP.md"],
    "decisions": ["Keep compaction-core API unchanged"],
    "confidence": "self_report_only"
  }
}
```

Properties:

- captures intent and "I no longer need raw bytes" directly;
- can explain decisions and pending work better than a pure event ledger;
- costs output tokens and replay tokens unless kept out of the visible transcript;
- can be wrong about future needs, so it needs harness guardrails.

## Decision Rule

Do not let the model directly replace compaction policy. Let it declare a
retention contract:

| Declaration | Harness action |
| --- | --- |
| `verbatim` | keep raw view or protect path from projection |
| `semantic` + hash match + no recent re-read | compact raw view into semantic ledger |
| `routing` | keep only path/hash/shape; discard bulky payload aggressively |
| `disposable` | compact immediately, but count later re-read as calibration failure |
| missing / malformed | fall back to deterministic projection |

The harness remains the final arbiter because R2 showed model self-orientation is
task-dependent. E1-C did not fail because no summary text existed; it failed
because the agent lost progress semantics and then re-read compacted paths.

## Cost Model

There are two possible placements.

### In-Band JSON

The JSON block is emitted in assistant content and replayed to the API.

Cost terms:

```text
extra_output = json_tokens_per_turn * turns
extra_replay_input = json_tokens_per_turn * remaining_turns
extra_reasoning = any additional hidden/reasoning effort needed to produce it
```

Rough R2-scale estimate:

- Small block: 80-150 tokens.
- Useful block with views/decisions/pending: 200-500 tokens.
- E1-style run: 20-60 turns.

If emitted every turn, that is about:

- output: 1.6k-30k extra tokens;
- replay input: potentially tens to hundreds of thousands of extra prompt tokens
  across the run if left in the transcript.

That is often too expensive unless it prevents re-read loops or artifact failure.
For E1-C, preventing one missing artifact or 25 compacted-path re-reads could be
worth it. For R1-C, it is probably wasted overhead because churn was already zero.

### Sideband / Harness-Captured JSON

The model emits the JSON through a tool call, structured output channel, or
delimited sidecar that the harness captures and removes from the next visible
prompt, replacing it with a compact ledger line.

Cost terms:

```text
extra_output = json_tokens_when_emitted
extra_replay_input = compact ledger delta only
```

This is the preferred design. It lets the model speak its intent once, but keeps
the long-lived context deterministic and hash-addressed.

## DeepSeek Reasoning Replay

DeepSeek currently matters because taucode's R2 data depends on DeepSeek cache
telemetry.

Official DeepSeek docs distinguish two relevant modes:

- `deepseek-reasoner` multi-round docs say previous `reasoning_content` should be
  removed before the next request; including it can cause a 400.
- DeepSeek v4 thinking mode docs say `reasoning_content` is returned alongside
  content, and when a model performs a tool call, that assistant
  `reasoning_content` must be passed back in subsequent user-interaction turns.

The harness must therefore preserve provider-specific replay semantics before it
does any compaction. For DeepSeek v4 tool-call sessions, reasoning is not just
display text; it is part of the valid API transcript. If we add model-authored
work semantics, we should avoid mixing it into raw reasoning. Put it in an
explicit content block/tool result/sidecar that can be independently stored,
validated, compacted, and replayed.

## Comparison With Frontier Services

Frontier providers are converging on hybrid context management:

- Anthropic recommends server-side compaction for long conversations and
  agentic workflows. It detects threshold crossing, summarizes, creates a
  compaction block, and continues with compacted context.
- OpenAI Responses supports `context_management` server-side compaction and a
  standalone compact endpoint. The compaction item can be encrypted/opaque and
  is not meant to be human-interpretable.
- Anthropic context-engineering guidance separates whole-transcript compaction
  from tool-result clearing: clearing old tool results is a surgical client-side
  operation, while compaction summarizes broader conversation state.
- Frameworks and agents combine lightweight client pruning, recent-window
  protection, external memory, and provider compaction.

taucode's gap/opportunity:

- DeepSeek gives useful cache telemetry but no comparable public server-side
  compaction contract.
- taucode can approximate the missing contract client-side, but its differentiator
  should be auditability: path/hash provenance, artifact retention, and explicit
  calibration of model intent declarations.

The proposed JSON block is closest to frontier structured/anchored summaries,
but with one important change: the summary is not trusted by itself. It becomes
a signed-by-context declaration indexed by `path#hash`, and the harness measures
whether the model later contradicts it by re-reading.

## Design Consequence

The next view-based-context packet should not be "replace deterministic
projection with model JSON." It should be:

1. Add a model declaration channel for retention intent.
2. Store declarations in a `path#hash` semantic ledger.
3. Let harness compaction use declarations as hints.
4. Inject trust hints when a view is stale or contradicted.
5. Measure calibration:
   - declared disposable but re-read;
   - declared semantic-complete but re-read verbatim;
   - declared verbatim but never used;
   - semantic ledger prevented a re-read.

## Minimal Experiment

Outcome addendum: this proposal was refined by Fable in
`docs/arch-c3-design-2026-07-06.md`. The in-band C''' arm below remains useful
as a capture/calibration study, but it is no longer the preferred first policy
arm. If C'' is insufficient on E1, the next policy arm is C-SB: C'' plus a
sideband summarizer that writes author-tagged summaries into the same
`path#hash` ledger while keeping main-loop reasoning and transcript replay
stable. Provider-token accounting is binding for the net-flow decision.

Add an E1 C''' arm:

- C baseline: deterministic projection.
- C'': deterministic work anchor.
- C''': deterministic work anchor + model-declared JSON sidecar.

Success criteria:

- static acceptance approaches D;
- total cost stays below D;
- compacted-path re-read rate drops vs C;
- sidecar output tokens are less than the input/re-read cost they prevent;
- false declarations are visible as calibration failures.

## One Sentence

Current anchors are harness-authored facts; the next step is model-authored
intent, stored and checked by the same hash ledger, never blindly trusted.
