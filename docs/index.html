Deterministic Context Projection in an Agent Loop: An Auditable Account of the Compaction Tax

Working Paper · Draft v0.1 · 2026-07-07 Self-contained: all claims are backed by files in this repository. No external citations. Status: preliminary evidence (n=3 paired runs per cell). Open items are listed in §8.

Abstract

Coding agents accumulate conversation context faster than they consume it, and every serving stack now prices that accumulation in a specific way: input tokens that extend an unchanged prompt prefix are billed at roughly one tenth of the base rate, while any byte that breaks the prefix re-prices everything after it. Context compaction — replacing old conversation content with summaries — is therefore not merely a window-management technique; it is a transaction against a cache ledger, and the transaction has a price that the dominant implementation (LLM-generated summaries) makes impossible to audit.

This paper describes taucode (formerly ecode during the pi-experiment phase; name unified 2026-07 — τ = 2π), a small research harness built on the pi coding agent, that replaces LLM summarization with deterministic context projection: a pure function from message history to a smaller message history, applied only at send time, gated by a token threshold below which the payload is byte-identical to the unprojected one. Because the projection is deterministic and the raw history is never modified, every consequence — cache-hit dips and recoveries, forced re-reads, lost task state — becomes attributable and measurable. We use this property to decompose the compaction tax into three separately measurable terms (summary production cost, prefix-break cost, and semantic-loss cost), and to run paired four-arm experiments against a provider (DeepSeek) that exposes per-request cache telemetry.

The headline result is deliberately narrow: on refactor/code-production tasks, deterministic projection reached the same acceptance and equal-or-better human-reviewed code quality at roughly one third of the baseline cost (n=3, medians). The negative results are equally load-bearing: on exploration tasks the same mechanism failed to preserve task-progress semantics and produced the worst run in the dataset; a semantic-anchor repair was tested against a token-matched placebo and its completion claim did not survive. We report both, the audit machinery that caught two of our own wrong conclusions before publication, and a roadmap of gated follow-ups.

1. Introduction: the economics, from first principles

1.1 What an agent turn pays for

A coding agent is a loop: the harness sends the accumulated conversation (system prompt, user messages, assistant messages, tool calls, tool results) to a model, the model replies with text and/or tool calls, the harness executes the tools and appends the results, and the loop repeats. Three facts about this loop generate everything else in this paper.

Fact 1: context grows monotonically, and most of it is tool payload. Each turn appends the full text of files read, command output, and the full arguments of write/edit calls. In our transcripts, old tool payloads dominate context size within ten to twenty turns.

Fact 2: providers price prefix stability, not just size. Serving stacks reuse the attention KV state of a request prefix that is byte-identical to a previous request. Providers pass this saving on: cached input tokens are billed at roughly 0.1× the uncached rate (we use DeepSeek's pricing and its per-request telemetry fields, e.g. prompt_cache_hit_tokens, throughout). The corollary is strict: the cost of a turn is not a function of context length alone; it is a function of how many of those bytes extend an unchanged prefix. An agent that grows its context append-only pays the expensive rate once per byte and the cheap rate ever after.

Fact 3: any compaction is a prefix regime change. Rewriting history — however cleverly — changes bytes early in the prompt. Every subsequent token of the request is then uncached. Compaction therefore never "saves tokens" in a single step; it trades a one-time cache break for a shorter prefix whose future turns are cheaper. Whether the trade wins depends on how long the session continues after the break and on what the summary destroyed.

This yields the cost inequality that a compaction scheme must satisfy to be worth running (first stated in docs/reports/r2-turn-interaction-retro-2026-07-06.md):

saved payload cost  >  cache-discontinuity cost + extra-turn cost + re-read cost

The right-hand side is the compaction tax. We decompose it into three terms:

  • T1 — summary production cost: tokens spent generating the summary itself (an LLM summarizer pays output-token prices; a deterministic function pays zero);
  • T2 — prefix-break cost: the one-time loss of the cache discount on the rewritten span and everything after it;
  • T3 — semantic-loss cost: the downstream behavior induced by information the summary destroyed — re-reading files whose contents were summarized away, wandering trajectories, or failing to produce the required artifact at all.

T3 is the largest and least discussed term. It never appears in a token-bill comparison, because its worst case is not a bigger bill — it is a completed-looking session with a missing artifact.

1.2 Why the dominant implementation cannot be audited

The standard compaction mechanism is to ask a model to summarize the conversation when the window fills. This has two structural problems, independent of summary quality.

First, nondeterminism destroys attribution. If the same session state can produce different summaries on different runs, then any downstream failure — a re-read burst, a lost requirement — cannot be attributed to a specific, inspectable transformation. The experiment "did compaction cause this?" has no stable treatment.

Second, the summary rewrites history in place. Once the original messages are replaced in the session record, the counterfactual (what the model would have seen without compaction) is gone, and so is the ability to audit what was lost.

The design hypothesis of this project is that both problems are removable at once: make the transformation a pure, deterministic function applied only to the outgoing payload, leave the persisted history untouched, and the entire causal chain — which bytes changed, what the cache did in response, what the model did in response to that — becomes measurable from logs.

1.3 The claim, stated as an existence proposition

We do not claim deterministic projection is universally better. The program is an existence proposition with a boundary-mapping agenda:

∃ (workload, threshold):  task semantics preserved (acceptance not degraded)
                          ∧ net cost < append-only baseline

Section 5 exhibits one proven point (refactor workload × 32k-token threshold). Section 6 maps where the proposition fails and why. Section 8 lists the sweeps that turn the point into a region.

2. Design principles

Four invariants govern the implementation. Each is enforced by tests, not by convention.

P1 — Determinism: same state, same bytes. The projection is a pure function of the message array and its configuration. Given the same input it renders the same output, byte for byte. This is what makes the projected prefix cacheable again after the one-time break (T2 is paid once, not per turn), and what makes failures attributable.

P2 — Identity below threshold. Below a configured token estimate (compactAfterInputTokens, default 32,000) the projection returns the same array reference, unchanged. Two consequences: sessions that never grow large pay zero — the mechanism can be permanently installed without a tax on small sessions; and the prefix of a warm session is never perturbed early, when cache value is highest. The same discipline applies within a projected send: if no message qualified for compaction, the identity is returned rather than a new-but-equal array (extensions/deterministic-compaction/src/projection.ts:99-103).

P3 — Raw history is append-only; projection exists only at send time. The transformation is applied in the harness's context hook, which shapes the outgoing request payload and is never persisted (verified against the host harness in docs/g0-survey.md, Item 3). The session file always contains the full, unmodified history. Auditing "what was hidden from the model at turn N" is a replay of a pure function, not an archaeology of overwritten records.

P4 — Provenance by content hash. Summaries and hints never assert facts about file state in prose; they carry content hashes and diffstats computed at tool-execution time. Anything the model is told about the past is mechanically checkable against the workspace.

Two further disciplines follow from P1–P3 and are worth naming because they exclude a known failure class. Idempotency: a projected summary is smaller than every eligibility threshold, so projecting twice is a no-op — the hook can run on every send without compounding. No silent prefix perturbation: every volatile addition (trust hints, anchors; §3.4–3.5) is appended strictly at the tail of the outgoing payload, after the stable prefix, so the cached span is never broken by advisory content.

3. System

3.1 Host: a thin loop with explicit seams

taucode is built on pi (the pi-mono coding agent), chosen precisely because it is a thin loop: its context assembly is small enough to be fully observable, and it exposes two extension seams that map exactly onto the design:

  • Seam A — a context hook that runs before every LLM call and may transform the outgoing message array; its return affects only the send payload and is never persisted.
  • Seam B — a session_before_compact hook that may replace pi's native LLM-summary compaction with a caller-supplied entry (kept optional and off by default in taucode; the native summarizer is retained as an experiment arm, not deleted).

The fork discipline is strict: pi/ carries zero diff against upstream; everything lives in an out-of-tree extension (extensions/deterministic-compaction/) and a standalone package (packages/compaction-core/). The extension is loaded by path (pi --extension .../src/extension.ts) and carries its mechanisms behind environment flags, all default-off except the projection itself.

3.2 The projection core: messages → messages

packages/compaction-core/ (~1,400 lines, no dependency on any harness code) implements the pure function. Its entry point:

compactCodeProductions(
  messages: Message[],
  options?: Partial<CompactionOptions>,   // keepRecentAssistantMessages=3,
                                          // minArgTokens=800, minResultTokens=200
  injection?: CompactionInjection,        // five seams, all defaulting to
): CompactionResult                       // original behavior

The function walks the message array once (O(n)), pairs tool calls with their results, and replaces two kinds of payload with typed, deterministic summaries:

  • Old write/edit tool-call arguments become a CodeProductionSummary: {path, chars, lines, head, tail, result, hash?, diffstat?} — the head/tail preserve a five-line preview; the hash and diffstat (when a ledger is injected, §3.4) give the summary provenance.
  • Old read/bash/search/find results become per-tool summaries that keep the locator and drop the content: a read summary keeps {path, lines, chars, hash?} plus the fixed instruction "Re-run read if exact content is needed"; bash keeps {command, exitCode, stderrLines}; search keeps {pattern, matchCount, fileCount, preview}.

Protection rules bound the blast radius: the last keepRecentAssistantMessages assistant turns are never touched; error results are never compacted; small payloads below the token floors are never compacted; a summary that would not be strictly smaller than its source is discarded. Every replacement is recorded in a diffs array ({turn, toolName, path, rawTokens, compactedTokens, tokensSaved}), which is the observability substrate for everything in §4.

Five injection seams (strategy set, tool-name matcher, path/hash extractor, per-path summary metadata, protected paths) let a harness adapt the core without forking it; every seam defaults to the original behavior, and the no-injection output is pinned byte-exact by tests.

3.3 Hybrid gating at the send boundary

The extension wires the core into Seam A (extensions/deterministic-compaction/src/projection.ts):

function projectContext(messages, config): ProjectionOutcome {
  const rawTokens = estimateAgentTokens(messages);      // pi's own estimator
  if (rawTokens < config.compactAfterInputTokens) {
    return { messages, projected: false, rawTokens };   // identity (P2)
  }
  const compaction = compactCodeProductions(...);
  if (compaction.compactedCount === 0) {
    return { messages, projected: false, rawTokens };   // identity again
  }
  return { messages: projected, projected: true, rawTokens, compaction };
}

The gate deliberately reuses the host's token estimator rather than a bespoke one: the trigger only needs to be consistent, while cost accounting uses provider- reported usage (a lesson recorded as a metric-unit failure class, §6.4). Below the gate, the send payload is byte-identical and the provider's prefix cache absorbs the whole history at the discounted rate. Above it, the projection produces a new — but deterministic, hence re-cacheable — prefix. This is the mechanism behind the transition shape measured in §5.3: one dip, then recovery.

3.4 The trust protocol: hashes against confabulation

Compacting a read result creates a hazard: the model may "remember" file content that has since changed, and a summary that merely says "you read this file" does not tell it whether that memory is current. The trust protocol makes staleness mechanical.

A session-scoped ledger records, per path, the content hash and diffstat of the latest read/edit (trust-ledger.ts). At send time, staleViewHints (trust-hint.ts:30) scans the outgoing messages for read views whose content hash differs from the ledger's current hash for paths the model itself has since edited, and emits at most one line per path:

[stale-view] src/config.ts: view from a1b2c3d4 predates your edit at turn 12
(now e5f6a7b8); re-read only if you need current content.

The hint is appended as a volatile tail message — after the stable prefix, never persisted, replaced (not accumulated) each turn. The wording is deliberately non-prescriptive ("re-read only if you need"): the protocol supplies verifiable temporal facts and leaves the decision to the model.

Field validation (§5.4) shows a production-tier model using these hints exactly as designed, including the hard case: distinguishing "stale because I edited it" (hash mismatch, re-read) from "current because I was the last editor" (hash match, no re-read).

3.5 Semantic anchors: the same channel, for task state

The exploration failure mode (§6.1) motivated a second tail block: a deterministic, LLM-free digest of work state — files read (path + hash + turn), edits made (path + diffstat + hash, failures recorded as failures), tests run (command + parsed result), and pending acceptance targets — accumulated from tool events by a WorkAnchor (anchor.ts), rendered only on projected turns. Its discipline mirrors the trust protocol: every line is a mechanically verifiable fact; no evaluative language ("almost done") is permitted. Whether this block buys anything is an empirical question, and §6.2 reports the answer we got, including the placebo control that dismantled our first interpretation.

3.6 Observability

Every mechanism reports into the same surfaces: per-turn JSONL rows (projected flag, per-path compaction diffs, provider usage including cache-hit tokens and reasoning tokens, re-read counters), a TUI gate widget showing threshold distance and trigger markers, a cache-hit trace, and ambient telemetry rows for ordinary (non-experiment) sessions. Machine-readable tail evidence blocks record exactly which volatile block (anchor, placebo, nudge, substitution) was present on which turn, so post-run audit can distinguish "tail rendered but ignored" from "tail absent" (tail-evidence.ts; requirement OBS-TAIL, §6.2). The extension carries 137 unit tests; the projection's identity and byte-stability properties are pinned by fixtures.

4. Measurement methodology

The harness exists to make one kind of sentence possible: "arm X differed from arm Y by Z, and here is the artifact trail." This section describes the controls that license such sentences at all.

4.1 Paired arms on frozen workspaces

The unit of measurement is a packet: a task prompt plus a workspace snapshot plus mechanical acceptance criteria. Three packets span the hypothesized regimes:

  • R1 — refactor/code-production: read large sources once, produce new modules;
  • E1 — exploration: survey a codebase and produce a required artifact (SUBSYSTEM-MAP.md) plus answers;
  • D1 — direct transform: convert data where exact original text may be needed after compaction (a designed negative probe).

Each packet runs under four arms: A (baseline: no compaction of any kind), B′ (pi's native LLM-summary compaction at a comparable window), C (deterministic projection, Seam A), D (hybrid: projection plus Seam B checkpoint). Workspaces are materialized from snapshots whose manifest hash is verified per run — all arms start from byte-identical trees. Arm order is re-randomized per repeat and interleaved across packets to prevent provider time-of-day effects from aliasing into arm effects. Model, provider, thresholds, and flags are recorded in a per-run manifest; n=3 repeats per cell.

Cost is computed from provider-reported usage in the unit uncached_input + 0.1 × cached_input (thousands), reflecting the actual pricing asymmetry; medians are reported with full lists alongside.

4.2 Codified judgment gates

Two failure classes are checked by pure functions over run summaries (experiments/lib/gates.ts), not by reviewer discretion:

  • invalid — the arm's mechanism never engaged (a projection arm whose threshold was never crossed; a native arm where the summarizer never fired). Deltas against such runs are meaningless and are excluded before interpretation.
  • suspicious — false savings: total tokens went down relative to baseline while churn went up (re-read count or compacted-path re-read rate increased). This encodes the project's founding negative result — token savings that are actually the model re-buying destroyed information — as a mechanical flag.

Both gates return the numbers that fired them and refuse to editorialize. Their fixture tests are part of the repository's test suite.

4.3 Controls above the gates

Three further disciplines proved decisive. Placebo tails: when testing whether an informative tail block (the anchor) helps, a token-matched but content-free tail is run as its own arm — pre-registered before the comparison, not improvised after (placebo-tail.ts). Blind artifact review: acceptance scripts bound quality from below; a human review of produced code, blinded to arm labels, is the actual quality instrument (§5.2). Verdict separation: run reports are evidence; a single verdict document (docs/stage-verdict-2026-07.md) holds the claims and their allowed phrasings, and downgrades every report to an appendix. Nothing is citable twice.

4.4 What the discipline caught

The methodology section would be decoration if it had never fired. It fired twice.

Round 1 produced a striking "projection causes 2.5× turn inflation" finding. The gates flagged half of round 1 as invalid (mechanisms not engaged), the finding was held out of the verdict, and round 2 (n=3) reversed it outright: the same arm ran at median 18 turns versus baseline 36. The retraction is recorded in docs/reports/r2-verdict.md (ruling 1).

Later, the semantic anchor's completion effect (§6.2) was pre-registered against a placebo; the placebo matched it, and the intended headline claim was withdrawn before it was ever made public. Both events are treated as products of the harness, not embarrassments to it: the audit trail is the deliverable.

5. Results

All numbers below are from paired real-provider runs (DeepSeek v4-flash via pi), 27 core runs plus 9 add-on runs, n=3 per cell, medians with full lists in the source reports (docs/reports/r2-verdict.md, r2-turn-interaction-retro-2026-07-06.md). Claims are stated with the boundary they were measured under and none beyond it.

5.1 The proven point: refactor × deterministic projection

Packet R1 (refactor) A (baseline) B′ (native) C (projection) D (hybrid)
cost, median [list] 1342 [492, 1342, 1744] 601 [433, 601, 762] 449 [401, 449, 857] 879
static acceptance 4/4 ×3 4/4 ×3 4/4 ×3 4/4 ×3
compacted-path re-reads 0 0 0 0, 2, 3

C's median cost is 33% of baseline with acceptance intact across all repeats, and — the mechanistically decisive column — zero re-reads of compacted paths: nothing the projection destroyed was ever needed again. The first-principles explanation is that refactor work converts read payloads into workspace edits; after the conversion, the durable state is the workspace, and the old bytes in context are pure debt. This is the workload the mechanism was designed for, and it behaves as designed.

5.2 Quality, bounded from above the scripts

Static checks bound quality from below. A blinded per-file human review of all three R1-C runs (review-R1-C-code-quality.md, summarized in the R2 verdict addendum) upgraded the claim: the cheapest run (365k tokens, 17 turns) was also judged the best code (clean encapsulation, no TODOs; sole blemish an export *), and the most expensive run was judged the worst. The cost claim is therefore cost-and-quality confirmed at n=3, not cost-at-quality's-expense.

The review produced a side finding we now treat as the first data point of a hypothesis: trajectory wandering and code degradation moved together — the worst-quality run was also the most expensive one. If trajectory length is a usable quality proxy, cost and quality are not a trade-off frontier at this scale but two projections of the same variable (whether the model stayed oriented). This is promoted to a measurement target, not asserted (§8).

5.3 The cache transition curve

Because the provider reports per-request cache hits, the T2 term is directly observable. The canonical good shape, from run 19-r3-G2-R1-C.jsonl:

turn payload input tokens cache-read tokens
10 raw 28,042 30,336
11 first projected 9,620 2,304
13 projected 14,825 13,312
15 projected 15,412 13,312
17 projected 16,109 20,608

One dip at the regime change, then recovery as the deterministic rendering holds byte-stable — exactly the trade described in §1.1: a one-time break purchasing a shorter, once-again-cached prefix. The counter-example run 14-r2-G2-E1-C shows why this curve is subordinate to semantics: its cache also recovered, and the run still produced no artifact. A cheap stable prefix is worthless to a model that no longer knows what it was doing — which is the bridge to §6.

5.4 The trust protocol in the field

A live flag-on session (docs/reports/v2-tp-wild-field-run.md) validated the stale-view mechanism end to end: no false positives when views were current; hint fired on a genuinely stale read and the model re-read before answering (correctly); and in the multi-file stress round, the model re-read only files where its view hash mismatched the ledger, declining to re-read files where it was itself the last editor. Cache-hit rate was unaffected by the tail append (the observed drop was attributable to re-read tokens, not prefix breakage). The protocol's design bet — that mechanical temporal facts, not prose confidence, are what a model can act on correctly — held under production conditions.

5.5 Self-application

The blind review in §5.2 was itself conducted inside a taucode session: the audit work ran with projection live, saving 56,780 tokens (67%) at a 98.1% cache-hit rate. We flag this as dogfooding, not as controlled evidence; its value is that the tooling's own paper trail was produced under the tooling.

6. Negative results

Each subsection is a claim we either expected to make and cannot, or a failure whose mechanism is now part of the design. The taxonomy labels (F1–F4) come from the turn-level retro and now have golden fixtures in the test plan.

6.1 F1 — work-semantic loss: projection fails exploration

Packet E1 (exploration) A B′ C D
cost, median 1527 1287 1616 2445
static acceptance 0/5, 5/5, 5/5 4/5, 5/5, 5/5 4/5, 0/5, 4/5 5/5 ×3
compacted-path re-read rate 0 0 .231, .211, .625 .150, .333, .174

On exploration, projection's cost advantage vanishes (1616 vs 1527) and its completion is unstable. The hard failure (14-r2-G2-E1-C): twelve projected turns, zero artifact files, the required SUBSYSTEM-MAP.md never produced, re-read bursts concentrated on compacted paths. The mechanism is legible in the logs: projection preserved file evidence (paths, hashes, previews) but destroyed progress semantics — what has been mapped, what remains, what artifact is owed. When the model lost orientation it re-opened compacted files (the re-read rate column is this signal) or, in the worst case, read and reasoned in circles without ever crossing the artifact-production boundary.

Honesty requires the baseline column too: A also failed once (0/5). Losing the thread on exploration tasks is partly task nature, not purely a compaction pathology. But the paired shape stands: what exploration needs preserved is task state, and extractive summaries do not carry it. Meanwhile arm D completed 6/6 across two rounds (the dataset's only such cell) at ~1.6× baseline cost — its checkpoint keeps a persistent progress digest in context. The verdict phrasing is deliberate: hybrid+checkpoint buys completion on exploration, not cost.

6.2 The anchor's completion claim died by placebo

The semantic anchor (§3.5) was built as the direct, LLM-free response to F1: if the missing ingredient is progress state, restate it deterministically on every projected turn. First results looked confirmatory. The controlled comparison (Branch B, with OBS-TAIL evidence on every turn) says otherwise:

E1 arm (n=3) completion median turns median total tokens
C″ (anchor) 2/3 21 677,535
C-SB (anchor + sideband summaries) 1/3 21 709,726
C+PL (token-matched placebo tail) 3/3 28 882,010

A content-free tail block of matched size completed the task more reliably than the informative anchor. The ruling (R14 in the design log) splits the variable the first interpretation had conflated: the ritual buys completion; the content buys trajectory. Any same-shaped reminder at projected turns appears to re-orient the model toward finishing; the anchor's verifiable facts show up instead in the cost column (C″ was the cheapest E1 arm observed, one run at 19 turns / 538k) — shorter, less wandering trajectories, not higher completion. Two corollaries went into the roadmap rather than the narrative: a governed minimal nudge (the ungoverned variant completed 3/3 but at ~6× cost — 84 turns, 3.23M tokens — pure ritual with no content is an expensive superstition), and a calibration study, since "declared disposable but re-read" is now measurable per path.

6.3 F2 — churn hidden by passing checks: the negative probe that didn't bite

D1, the designed negative zone (exact-text dependency under an aggressive 8k threshold), failed to produce the expected failure: C passed 4/4 ×3 with median cost below baseline. But the logs show why it survived, and the why is a warning: compacted-path re-read rates ran as high as .917 and .926 — the model repeatedly asked for exact content, the harness repeatedly compressed it away, and re-reading repaired the loss each time. The probe was survivable by paying T3 in small installments. We record this as "the current probe is too weak," not "the negative zone is empty"; a stronger verbatim-dependency probe is queued (§8). The general lesson is methodological: acceptance scripts alone would have called this a clean pass; the churn counters are what kept it honest.

6.4 Smaller instruments that failed and were kept

Sideband summaries (C-SB) — off-thread LLM summarization of tool results — violated their own economics in the current implementation: one run spent 186 sideband calls / 158k output tokens to complete 0/5. The net-flow accounting (summaries must cost less than they save) is now a mock-level precondition for any rerun; the design is unproven, not disproven, and stays parked. Native-trigger nondeterminism (F3): pi's window-pressure compaction fired only 1/3 on the small packet, invalidating two B′ runs — validity gating on observed engagement is now standard. Metric-unit mismatch (F4): local token estimates and provider usage are close in direction but not per-turn comparable; all cost math now uses provider units, estimates only gate. In-band declarations: having the model declare work semantics in-channel costs ~+80 output and ~+66 reasoning tokens per turn (piggybacked, no turn inflation) — an affordable instrument for the calibration study, not a free one.

7. Scope and limitations

Everything above holds within, and only within, the following frame. The measurements cover loop-internal context economics: a thin harness whose entire context assembly is observable, one provider whose cache telemetry is exposed (DeepSeek), one model generation (v4-flash), projection thresholds between 8k and 48k tokens, three task packets, n=3 per cell. Medians and paired ranks are reported; no significance claims are made or licensed at this n. Frontier agent products assemble context outside the loop (retrieval, IDE state, hidden tool output) in ways that are not observable from where we stand; nothing here measures them. Model-generation dependence is a finding, not a nuisance: which failure modes a model exhibits under compaction (and how well its self-reports calibrate) shifts with training, which is precisely why the calibration rate is proposed as a capability measurement rather than assumed constant. Extrapolation to million-token windows is a mechanism argument (§1.1 survives any window size; the discount asymmetry and T3 do not disappear), not a data claim. The small scale is a choice, not a concession: 8k–48k on a cache-observable provider is currently the only place every line of the compaction-tax ledger can be independently audited.

One further honesty note: the cost unit prices cached input at 0.1× but ignores provider-side variation in cache retention and any charge model where cached spans still occupy window; results should port across providers in shape, not in constants.

8. Roadmap

Every open item carries its gate; none is a promise.

  1. SWEEP-R2 — threshold curve: R1-C at 4k/16k/64k and D1-C at 4k, six runs, same audit. Gate for: the sweet-spot boundary, and for upgrading §5.1 from a point to a curve.
  2. Stronger verbatim probe — a task where re-reading cannot repair exactness loss (or is made expensive). Gate for: any claim about the direct-transform row of the dispatch policy.
  3. Second trajectory×quality data point — required before the §5.2 proxy hypothesis is used anywhere.
  4. Governed nudge (NUDGE-GOV) — the placebo result implies a minimal ritual tail; ungoverned it cost 6×. Injection cap plus stop-on-empty-pending. Not scheduled until Branch-B follow-ups clear.
  5. Calibration study (C‴) — measure "declared disposable but re-read" as a per-model calibration rate; the declaration tax numbers (§6.4) are in hand. This is the paper-axis item: the rate is a capability contour, usable across model generations.
  6. Ledger continuation (WS-6) — restore work-semantic state across sessions from the persisted ledger. Gated on dogfooding evidence and retention rules.
  7. Provider-compaction arm (E) — when a cache-observable provider exposes a server-side compaction API, compare it as an arm rather than rewriting the thesis around it.
  8. Dispatch policy hardening — the R2 mapping (refactor → projection; exploration/completion-first → checkpoint; exploration/cost-first → native or baseline; direct-transform → no special handling at current probe strength) ships as a recommendation layer only, until it survives held-out task families.

The stop conditions are as much a part of the roadmap as the queue: a mechanism that lowers tokens but raises artifact failure stops; a mechanism that wins under only one prompt shape stays experimental; a provider feature that obsoletes a local rule retires the rule and keeps the measurement harness.

9. A note on how this was built

The repository's code was written by LLM coding agents working from self-contained task packets (specification, exact file list, acceptance criteria, prohibited zones), test-first, with the packets and their revision history preserved in the repo's goal ledger. Human involvement was deliberately concentrated at the judgment layer: parameter choices, quality review, gate rulings, and the decision of what may be claimed — none of which was delegated. Two operational incidents during the project (a summary-induced context loss affecting the supervising session itself, and a hard-zone violation by a weaker model) were captured, rolled back, and codified into the collaboration protocol rather than discarded. We note this both for provenance and because it is a small instance of the paper's subject: the workflow was itself an exercise in deciding which context to trust, which to verify, and which to deliberately forget.

智能体循环中的确定性上下文投影：压缩税的可审计账本

工作论文 · 草案 v0.1 · 2026-07-07 自包含：所有主张均由本仓库中的文件支持。无外部引用。 状态：初步证据（每组 n=3 次配对运行）。未决事项列于第 8 节。

摘要

智能体在编码过程中积累对话上下文的速度远快于其消耗速度，而现在的每个推理服务栈都以特定方式对这种积累进行计费：扩展未改变前缀的输入 Token 按基础费率的约十分之一计费，而任何破坏前缀的字节都会重新计算其后所有内容的价格。因此，上下文压缩（用摘要代替旧的对话内容）不仅是一种窗口管理技术，而且是对缓存账本的交易，且该交易具有主导实现（LLM 生成的摘要）无法审计的价格。

本文介绍了 taucode（pi 实验阶段曾用名 ecode；2026-07 统一命名，取 τ = 2π），这是一个基于 pi 编码智能体构建的微型研究框架，它将 LLM 摘要替换为确定性上下文投影（deterministic context projection）：这是一个从消息历史到较短消息历史的纯函数，仅在发送时应用，并受 Token 阈值限制，低于该阈值时发送的负载与未投影的负载完全一致。由于投影是确定性的，且原始历史从未被修改，因此每个后果——缓存命中率的下降与恢复、强制重读、丢失的任务状态——都变得可追溯和可衡量。我们利用这一特性将压缩税分解为三个可单独衡量的项（摘要生成成本、前缀中断成本和语义损失成本），并针对提供单次请求缓存遥测的云服务商（DeepSeek）运行了配对的四臂实验。

核心结果故意收窄：在重构/代码生成任务中，确定性投影达到了相同的采纳率，并在人工评估中获得了相同或更好的代码质量，而成本仅为基线的三分之一左右（n=3，中位数）。负面结果同样具有支撑作用：在探索性任务中，相同的机制未能保留任务进度语义，并产生了数据集中最差的运行结果；在针对 Token 匹配的安慰剂对照测试中，语义锚点修复的完成声明未能通过验证。我们报告了这两者、在发表前捕捉到我们自身两个错误结论的审计机制，以及后续门控研究的路线图。

1. 引言：基于第一性原理的经济学

1.1 智能体单次轮次支付的费用

编码智能体是一个循环：框架将累积的对话（系统提示词、用户消息、助手消息、工具调用、工具结果）发送给模型，模型回复文本和/或工具调用，框架执行工具并追加结果，循环往复。关于这个循环的三个事实引出了本文的其余所有内容。

事实 1：上下文呈单调增长，且大部分是工具载荷。 每一轮都会追加读取文件的全文、命令输出以及写入/编辑调用的完整参数。在我们的转录本中，旧的工具载荷在十到二十轮之内就占据了上下文的主要大小。

事实 2：服务商对前缀稳定性进行定价，而不仅仅是大小。 服务栈会重用与先前请求字节完全相同的请求前缀的 attention KV 状态。服务商将这种节省回馈给用户：缓存的输入 Token 按未缓存费率的约 0.1 倍计费（我们全程使用 DeepSeek 的定价及其单次请求遥测字段，例如 prompt_cache_hit_tokens）。其推论是严苛的：单次轮次的成本不仅是上下文长度的函数，也是这些字节中有多少扩展了未改变前缀的函数。 一个采用仅追加方式增长上下文的智能体，对每个字节仅支付一次昂贵的费率，之后便一直享受便宜的费率。

事实 3：任何压缩都是前缀规则的改变。 重写历史——无论多么聪明——都会改变 Prompt 早期的字节。随后请求中的每个 Token 都会变得未缓存。因此，压缩绝不会在单一步骤中“节省 Token”；它是用一次性的缓存中断换取更短的前缀，从而使未来的轮次更便宜。这种交易是否划算，取决于中断后会话持续了多久，以及摘要破坏了什么内容。

这得出了一个压缩方案必须满足的成本不等式，才值得运行（首次阐述于 docs/reports/r2-turn-interaction-retro-2026-07-06.md）：

节省的载荷成本  >  缓存不连续性成本 + 额外轮次成本 + 重读成本

等式的右侧即为压缩税。我们将其分解为三项：

  • T1 — 摘要生成成本：生成摘要本身所消耗的 Token（LLM 摘要器需要支付输出 Token 的价格；确定性函数则支付零成本）；
  • T2 — 前缀中断成本：重写区间及其后所有内容一次性失去缓存折扣的损失；
  • T3 — 语义损失成本：摘要破坏的信息所引起的下游行为——重新读取已被摘要过滤掉的文件内容、迷失的轨迹，或者根本无法生成所需的产物。

T3 是最大且最少被讨论的项。它从未出现在 Token 账单对比中，因为它的最坏情况不是更大的账单——而是一个看起来已完成但缺失了产物的会话。

1.2 为什么主导实现无法被审计

标准的压缩机制是在窗口填满时让模型总结对话。无论摘要质量如何，这都存在两个结构性问题。

首先，非确定性破坏了归因。如果相同的会话状态在不同的运行中会产生不同的摘要，那么任何下游失败——重读爆发、遗漏需求——都无法归因于特定的、可检查的转换。对于“压缩是否导致了这一结果？”这一实验，没有稳定的处理对照。

其次，摘要会就地重写历史。一旦会话记录中的原始消息被替换，反事实（如果模型在没有压缩的情况下会看到什么）就消失了，审计丢失内容的能力也随之消失。

本项目的核心设计假设是，这两个问题可以同时消除：将转换设计为仅应用于发送载荷的纯确定性函数，保持持久化的历史记录不受影响，这样整个因果链——哪些字节发生了变化、缓存做出了何种响应、模型又对此做出了何种反应——都可以从日志中进行衡量。

1.3 声明：作为存在性命题阐述

我们并不声称确定性投影普遍更好。该程序是一个具有边界映射议程的存在性命题：

∃ (工作负载, 阈值):  任务语义得以保留 (采纳率未降低)
                          ∧ 净成本 < 仅追加基线

第 5 节展示了一个已证实的点（重构工作负载 × 32k Token 阈值）。第 6 节映射了该命题在何处失败以及原因。第 8 节列出了将点扩展为区域的参数扫频。

2. 设计原则

四个不变量支配着实现。每一个都由测试强制执行，而非约定俗成。

P1 — 确定性：状态相同，字节相同。 投影是消息数组及其配置的纯函数。给定相同的输入，它会渲染出字节完全相同的输出。这就是为什么被投影的前缀在一次性中断后可以再次被缓存（T2 只支付一次，而不是每轮都付），也是为什么失败是可以归因的原因。

P2 — 低于阈值时保持原样。 在配置的 Token 估算值（compactAfterInputTokens，默认 32,000）以下，投影返回相同的数组引用而不做任何修改。两个后果：从未变大的会话支付零成本——该机制可以永久安装，而不会对小型会话征税；并且在会话发热的早期（此时缓存价值最高），前缀绝不会受到扰动。在投影发送内部也应用相同的纪律：如果没有消息符合压缩条件，则返回恒等引用，而不是一个相同但新建的数组（extensions/deterministic-compaction/src/projection.ts:99-103）。

P3 — 原始历史仅追加；投影仅在发送时存在。 转换在框架的 context 钩子中应用，该钩子塑造了发送的请求载荷，并且从未被持久化（针对宿主框架的验证见 docs/g0-survey.md 第 3 项）。会话文件始终包含完整的、未经修改的历史记录。审计“在第 N 轮对模型隐藏了什么”是对纯函数的重放，而不是对被重写记录的考古。

P4 — 通过内容哈希追溯源头。 摘要和提示绝不在散文中声称关于文件状态的事实；它们携带在工具执行时计算的内容哈希和 diff 统计信息。模型被告知的关于过去的任何事情，都可以在工作区中进行机械式验证。

另外两个准则源自 P1–P3，值得提及，因为它们排除了一类已知的失败模式。幂等性：投影摘要小于每个符合条件的阈值，因此投影两次是无操作（no-op）——钩子可以在每次发送时运行而不会产生累积效应。无静默前缀扰动：每个易失性添加（信任提示、锚点；第 3.4–3.5 节）都严格追加在发送载荷的末尾，即在稳定前缀之后，因此缓存的区间绝不会被建议性内容破坏。

3. 系统

3.1 宿主：具有显式接缝的微型循环

taucode 构建在 pi（pi-mono 编码智能体）之上，选择它正是因为它是一个微型循环：其上下文组装足够小，完全可观测，并且它暴露了两个与设计完全契合的扩展接缝：

  • 接缝 A — 一个在每次 LLM 调用之前运行的 context 钩子，可以转换 outgoing 消息数组；其返回值仅影响发送载荷，绝不持久化。
  • 接缝 B — 一个 session_before_compact 钩子，可以用调用者提供的条目替换 pi 原生的 LLM 摘要压缩（在 taucode 中保持可选且默认关闭；原生摘要器作为实验分支保留，未被删除）。

分支规范非常严格：pi/ 与上游相比没有任何 diff；所有内容都存在于树外扩展（extensions/deterministic-compaction/）和独立包（packages/compaction-core/）中。扩展通过路径加载（pi --extension .../src/extension.ts），并将其机制置于环境变量标志之后，除投影本身外，所有标志默认关闭。

3.2 投影核心：messages → messages

packages/compaction-core/（约 1,400 行，不依赖于任何框架代码）实现了该纯函数。其入口点为：

compactCodeProductions(
  messages: Message[],
  options?: Partial<CompactionOptions>,   // keepRecentAssistantMessages=3,
                                          // minArgTokens=800, minResultTokens=200
  injection?: CompactionInjection,        // 五个接缝，全部默认为原始行为
): CompactionResult

该函数遍历消息数组一次（O(n)），将工具调用与其结果配对，并用类型化的、确定性的摘要替换两种载荷：

  • 旧的写入/编辑工具调用参数变为 CodeProductionSummary： {path, chars, lines, head, tail, result, hash?, diffstat?} —— head/tail 保留了五行预览；哈希和 diffstat（当注入账本时，第 3.4 节）为摘要提供了来源。
  • 旧的读取/bash/搜索/查找结果变为每个工具的摘要，保留定位符并丢弃内容：read 摘要保留 {path, lines, chars, hash?} 以及固定指令“如果需要确切内容，请重新运行 read”；bash 保留 {command, exitCode, stderrLines}；search 保留 {pattern, matchCount, fileCount, preview}。

保护规则限制了影响范围：最后 keepRecentAssistantMessages 轮助手回合绝不会被触及；错误结果绝不压缩；低于 Token 下限的小载荷绝不压缩；不会比其源内容更小的摘要将被丢弃。每次替换都会记录在 diffs 数组中（{turn, toolName, path, rawTokens, compactedTokens, tokensSaved}），这是第 4 节中所有内容的可观测性基质。

通过五个注入接缝（策略集、工具名称匹配器、路径/哈希提取器、单路径摘要元数据、受保护路径），框架可以在不分叉核心的情况下对其进行适配；每个接缝默认使用原始行为，且无注入输出通过测试进行字节精确的固定。

3.3 发送边界处的混合门控

扩展将核心连接到接缝 A（extensions/deterministic-compaction/src/projection.ts）：

function projectContext(messages, config): ProjectionOutcome {
  const rawTokens = estimateAgentTokens(messages);      // pi 自己的估算器
  if (rawTokens < config.compactAfterInputTokens) {
    return { messages, projected: false, rawTokens };   // 恒等 (P2)
  }
  const compaction = compactCodeProductions(...);
  if (compaction.compactedCount === 0) {
    return { messages, projected: false, rawTokens };   // 再次恒等
  }
  return { messages: projected, projected: true, rawTokens, compaction };
}

该门控故意重用了宿主的 Token 估算器，而不是定制的估算器：触发器只需要保持一致即可，而成本核算使用服务商报告的用量（这是从度量单位失效类别中汲取的教训，第 6.4 节）。在门控之下，发送载荷完全一致，服务商的前缀缓存以折扣费率吸收整个历史记录。在门控之上，投影产生一个新的——但由于是确定性的，因此可以重新缓存——前缀。这就是第 5.3 节中测得的过渡形状背后的机制：一次下降，然后恢复。

3.4 信任协议：用哈希对抗胡言乱语

压缩 read 结果会带来风险：模型可能会“记住”此后已更改的文件内容，而仅显示“您读取了此文件”的摘要无法告诉它该记忆是否为最新。信任协议使陈旧性判断变得机械化。

会话范围的账本记录了每个路径最新读取/编辑的内容哈希和 diff 统计信息（trust-ledger.ts）。在发送时，staleViewHints（trust-hint.ts:30）扫描 outgoing 消息，寻找其内容哈希与账本当前哈希不同的 read 视图（针对模型自身此后编辑过的路径），并为每个路径最多输出一行提示：

[stale-view] src/config.ts: view from a1b2c3d4 predates your edit at turn 12
(now e5f6a7b8); re-read only if you need current content.

提示作为易失性尾部消息追加——在稳定前缀之后，绝不持久化，每轮都会被替换（而不是累加）。其措辞故意是非指令性的（“仅在需要时重读”）：该协议提供可验证的临时事实，并将决定权留给模型。

现场验证（第 5.4 节）显示，生产级模型完全按照设计使用这些提示，包括困难情况：区分“因我编辑而陈旧”（哈希不匹配，重读）与“因我是最后的编辑者而最新”（哈希匹配，不重读）。

3.5 语义锚点：相同通道中的任务状态

探索失败模式（第 6.1 节）促成了第二个尾部块的产生：一个确定性的、无 LLM 参与的工作状态摘要——读取的文件（路径 + 哈希 + 轮次）、做出的编辑（路径 + diffstat + 哈希，失败记录为失败）、运行的测试（命令 + 解析后的结果）以及悬而未决的验收目标——这些信息由 WorkAnchor（anchor.ts）从工具事件中收集，仅在投影轮次中渲染。其规范镜像了信任协议：每一行都是机械可验证的事实；不允许使用任何评估性语言（例如“即将完成”）。这个块是否能带来收益是一个实证问题，第 6.2 节报告了我们得到的答案，包括解构了我们最初解释的安慰剂对照组。

3.6 可观测性

每个机制都向相同的界面报告：逐轮 JSONL 行（投影标志、单路径压缩 diff、服务商用量（包括缓存命中 Token 和推理 Token）、重读计数器）、一个显示阈值距离和触发标记的 TUI 门控挂件、一个缓存命中追踪，以及用于普通（非实验）会话的环境遥测行。机器可读的尾部证据块精确记录了哪一轮存在哪个易失性块（锚点、安慰剂、微调、替换），以便在运行后审计时能够区分“尾部已渲染但被忽略”与“尾部缺失”（tail-evidence.ts；要求 OBS-TAIL，第 6.2 节）。扩展包含 137 个单元测试；投影的恒等和字节稳定性属性已通过固定装置锁死。

4. 测量方法论

该测试框架的存在是为了让以下陈述成为可能：“分支 X 与分支 Y 在 Z 上的差异如下，这是对应的产物轨迹。” 本节描述了使此类陈述具有可信度的控制措施。

4.1 冻结工作区上的配对分支

测量的单位是数据包 (packet)：任务提示词加上工作区快照，再加上机械化验收标准。三个数据包涵盖了假设的运作状态：

  • R1 — 重构/代码生成：一次性读取大型源文件，生成新模块；
  • E1 — 探索：调查代码库并生成所需的产物（SUBSYSTEM-MAP.md）以及解答；
  • D1 — 直接转换：转换数据，在压缩后可能需要确切的原始文本（设计好的负面探测）。

每个数据包在四个分支下运行：A（基线：不进行任何形式的压缩），B′（pi 在同等窗口下的原生 LLM 摘要压缩），C（确定性投影，接缝 A），D（混合型：投影加上接缝 B 检查点）。工作区是从清单哈希经过逐次运行验证的快照中实例化的——所有分支都从字节相同的目录树开始。分支的顺序在每次重复时重新随机化，并在数据包之间交错进行，以防止服务商在一天中不同时间的性能波动影响分支效果。模型、服务商、阈值和标志都记录在每次运行的清单中；每个单元重复 n=3 次。

成本根据服务商报告的用量计算，单位为 uncached_input + 0.1 × cached_input（千个 Token），这反映了实际的计费不对称性；中位数与完整列表并排报告。

4.2 程式化的评估门控

运行摘要（experiments/lib/gates.ts）上的两个失效类别由纯函数检查，而非评审员酌情决定：

  • invalid — 分支的机制从未生效（阈值从未被跨越的投影分支；摘要器从未启动的原生分支）。这些运行的差异是没有意义的，并在解释之前被排除。
  • suspicious — 虚假节省：相对于基线，总 Token 数下降，但流失率上升（重读计数或压缩路径重读率增加）。这以机械标志的形式编码了项目创立之初的负面结果——即模型通过重新获取已被破坏的信息来换取 Token 的节省。

两个门控都只返回触发它们的数据，拒绝进行主观评论。它们的固定装置测试是仓库测试套件的一部分。

4.3 门控之上的控制

另外三项纪律被证明是决定性的。安慰剂尾部：在测试信息性尾部块（锚点）是否有帮助时，会运行一个 Token 大小匹配但内容为空的尾部作为其自身的分支——在对比前预先注册，而不是事后临时起意（placebo-tail.ts）。双盲产物评审：验收脚本从下方约束质量；对生成的代码进行不带分支标签的人工评审，是实际的质量评估工具（第 5.2 节）。结论分离：运行报告是证据；一份单独的结论文档（docs/stage-verdict-2026-07.md）保存了主张及其允许的措辞，并将每份报告降级为附录。任何内容都不能被引用两次。

4.4 规范所捕获的问题

如果这些规范从未发挥作用，那方法论章节就成了摆设。事实上，它发挥了两次作用。

第一轮实验产生了一个引人注目的结论：“投影导致了 2.5 倍的轮次膨胀”。然而，评估门控将第一轮的一半运行标记为无效（机制未生效），该发现被排除在结论之外，而第二轮（n=3）则完全逆转了这一结果：相同的分支以中位数 18 轮运行，而基线为 36 轮。这一撤回记录在 docs/reports/r2-verdict.md 中（裁决 1）。

后来，语义锚点的完成效果（第 6.2 节）针对安慰剂组进行了预注册；结果安慰剂组与它打平，预期的核心主张在公之于众之前就被撤回了。这两起事件都被视为该框架的产物，而不是它的尴尬：审计轨迹本身就是交付物。

5. 实验结果

以下所有数据均来自配对的真实服务商运行（通过 pi 运行 DeepSeek v4-flash），包括 27 次核心运行和 9 次附加运行，每个单元 n=3，中位数及完整列表参见源报告（docs/reports/r2-verdict.md，r2-turn-interaction-retro-2026-07-06.md）。主张仅在其被测量的边界内阐述，不作过度推演。

5.2 高于脚本约束的质量评估

静态检查从下方约束了质量。对所有三个 R1-C 运行进行不带标签的逐文件人工评审（review-R1-C-code-quality.md，总结在 R2 结论增补中）提升了这一主张：成本最低的运行（365k Token，17 轮）也被评为质量最佳的代码（干净的封装，无 TODO；唯一的瑕疵是 export *），而最昂贵的运行被评为最差的代码。因此，在 n=3 的水平上，成本主张被证实为“低成本兼具高质量”，而不是以牺牲质量为代价换取低成本。

评审产生了一个副发现，我们现在将其视为一个假设的第一个数据点：轨迹漂移与代码降级是同向运动的——质量最差的运行同时也是成本最高的运行。如果轨迹长度是一个可用的质量代理指标，那么在这种规模下，成本和质量就不是此消彼长的关系，而是同一变量的两个投影（即模型是否保持了方向感）。这被提升为一个衡量目标，而非既成断言（第 8 节）。

5.3 缓存过渡曲线

由于服务商报告了单次请求的缓存命中情况，因此 T2 项是直接可观测的。来自运行 19-r3-G2-R1-C.jsonl 的典型良好形态如下：

轮次 载荷 输入 Token 缓存读取 Token

11 首次投影 9,620 2,304

在规则改变时出现一次下降，然后随着确定性渲染保持字节稳定而恢复——这正是第 1.1 节中描述的交易：一次性中断换取更短、再次被缓存的前缀。反例运行 14-r2-G2-E1-C 展示了为什么这条曲线从属于语义：它的缓存虽然也恢复了，但该运行依然未能生成任何产物。对于一个不再知道自己在做什么的模型来说，一个便宜且稳定的前缀毫无价值——这正是通往第 6 节的桥梁。

5.4 现场的信任协议

一次开启标志的实地运行会话（docs/reports/v2-tp-wild-field-run.md）端到端验证了陈旧视图机制：当视图为最新时无误报；在一次真正陈旧的读取上触发了提示，模型在回答前重新读取了内容（表现正确）；并且在多文件压力测试轮中，模型仅重新读取了其视图哈希与账本不匹配的文件，而没有重新读取它本身作为最后编辑者的文件。缓存命中率未受尾部追加的影响（测得的下降可归因于重读 Token，而非前缀破坏）。该协议的设计赌注——模型能够正确应对的是机械的临时事实，而非散文式的确定性陈述——在生产条件下得到了证实。

5.5 自主应用

第 5.2 节中的双盲评审本身就是在 taucode 会话中进行的：审计工作在投影启用的情况下运行，在 98.1% 的缓存命中率下节省了 56,780 个 Token（67%）。我们将其归为“吃自家狗粮”（dogfooding），而不是受控的实验证据；它的价值在于，该工具自身的纸面记录就是在此工具下生成的。

6. 负面结果

每个子章节都是我们原本期望提出但无法提出的主张，或者是其机制现已成为设计一部分的失败案例。分类标签（F1–F4）来自轮次级回顾，目前在测试计划中拥有黄金测试夹具。

在探索任务中，投影的成本优势消失了（1616 对 1527），且其完成率不稳定。严重失败案例（14-r2-G2-E1-C）：运行了 12 个投影轮次，生成的产物文件为零，所需的 SUBSYSTEM-MAP.md 从未生成，重读爆发集中在压缩路径上。日志中的机制清晰可见：投影保留了文件证据（路径、哈希、预览），但破坏了进度语义——什么已被映射、还剩什么、欠缺什么产物。当模型失去方向时，它会重新打开已压缩的文件（重读率这一列就是该信号），或者在最坏的情况下，原地打转进行读取和推理，而从未跨越产物生成的边界。

诚实起见，基线列也应予以考虑：A 分支也失败过一次（0/5）。在探索性任务中迷失方向在一定程度上是任务性质使然，并非纯粹是压缩病理。但配对的规律依然成立：探索所需保留的是任务状态，而提取式摘要并不携带这些状态。与此同时，分支 D 在两轮中完成了 6/6（数据集中唯一的此类单元），成本约为基线的 1.6 倍——其检查点在上下文中保留了持久的进度摘要。结论的措辞是深思熟虑的：混合+检查点在探索中换取的是完成度，而不是成本节省。

6.2 锚点的完成主张在安慰剂对照下失效

语义锚点（第 3.5 节）是针对 F1 专门构建的、无 LLM 参与的直接响应：如果悬空要素是进度状态，则在每个投影轮次中以确定性方式重申它。初步结果看起来是证实性的。然而，受控对比（带有每轮 OBS-TAIL 证据的分支 B）给出了不同的结论：

E1 分支 (n=3) 完成率 中位数轮次 中位数总 Token 数
C″ (锚点) 2/3 21 677,535
C-SB (锚点 + 旁路摘要) 1/3 21 709,726

一个尺寸相同但内容为空的尾部块，比包含丰富信息的锚点更可靠地完成了任务。设计日志中的裁决（R14）分离了最初解释中所混淆的变量：仪式换来的是完成，内容换来的是轨迹。 在投影轮次中追加任何相同形状的提醒，似乎都能重新引导模型走向完成；而锚点提供的可验证事实则体现在成本一栏中（C″是观察到的最便宜的 E1 分支，一次运行为 19 轮 / 538k Token）——它缩短了轨迹，减少了无序漫游，而不是提高了完成率。两项推论被列入了路线图而不是叙事中：受控的极简提示（失控的变体完成了 3/3，但成本约为 6 倍——84 轮，3.23M Token——没有任何内容的纯仪式是一种昂贵的迷信），以及一项标定研究，因为“声明为可丢弃但被重读”的现象现在可以按路径进行测量。

6.3 F2 — 逃过检查的流失：未见成效的负面探测

D1，即设计的负面区域（在激进 8k 阈值下对确切文本的依赖），未能产生预期的失败：分支 C 以低于基线的中位数成本通过了 4/4 ×3。但日志显示了它得以存活的原因，而这一原因是一个警告：压缩路径的重读率高达 .917 和 .926——模型一再请求确切内容，框架一再将其压缩掉，而重读在每次都修复了这一损失。该探测之所以能存活，是通过分期支付 T3 达成的。我们将其记录为“当前的探测手段过于薄弱”，而非“负面区域为空”；一个更强的逐字依赖探测器已在排队中（第 8 节）。普遍的教训是方法论上的：单凭验收脚本会判定这是一个干净的通过；而流失计数器才是保持其诚实的工具。

6.4 失败但被保留的微型工具

旁路摘要 (C-SB)——对工具结果进行离线 LLM 摘要——在当前的实现中违背了其自身的经济学：一次运行消耗了 186 次旁路调用 / 158k 输出 Token，最终完成度为 0/5。净流出核算（摘要成本必须低于其节省的成本）目前已成为任何重新运行的模拟级前提条件；该设计尚未被证实，但也未被证伪，暂时搁置。原生触发的非确定性 (F3)：pi 的窗口压力压缩在小数据包上仅触发了 1/3，导致两次 B′ 运行无效——基于观察到的参与度进行有效性门控现已成为标准。度量单位失配 (F4)：本地 Token 估算与服务商实际用量方向接近，但无法在逐轮层面进行比较；所有成本数学计算现在均使用服务商单位，估算仅用于门控。带内声明：让模型在通道内声明工作语义每轮增加约 80 个输出 Token 和约 66 个推理 Token（搭载传输，无轮次膨胀）——这对于标定研究来说是一个负担得起的手段，但并非免费。

7. 范围与局限性

以上所有结论均在以下框架内成立，且仅在该框架内成立。测量涵盖了循环内部的上下文经济学：一个其整个上下文装配均可观测的微型框架，一个公开了缓存遥测的服务商（DeepSeek），一代模型（v4-flash），8k 至 48k Token 之间的投影阈值，三个任务数据包，每个单元 n=3。报告了中位数和配对排名；在此 n 值下不作任何显著性声明，亦不具备此类授权。前沿智能体产品在循环外部装配上下文（检索、IDE 状态、隐藏工具输出），其方式从我们所处的位置来看是不可观测的；这里的任何内容都无法衡量它们。模型世代依赖性是一项发现，而非干扰因素：模型在压缩下表现出何种失效模式（以及其自我报告的标定效果如何）会随着训练而改变，这正是将标定率作为能力指标来衡量、而非假定其恒定的原因。外推至百万 Token 窗口是一个机制层面的论点（第 1.1 节在任何窗口大小下均有效；折扣不对称性和 T3 并未消失），而非数据层面的断言。选择小规模是一个决定，而非妥协：在支持缓存观测的服务商上进行 8k–48k 的测试，是目前唯一可以对压缩税账本的每一行进行独立审计的地方。

另一个诚实性说明是，成本单位将缓存输入按 0.1 倍计费，但忽略了服务商端在缓存保留期方面的差异，以及缓存区间仍占用窗口的任何计费模型；实验结果在形态上应当跨服务商迁移，但在常数上则不然。

8. 路线图

每一个开放项都带有其门控条件，均非承诺。

  1. SWEEP-R2 — 阈值曲线：在 4k/16k/64k 下的 R1-C 以及在 4k 下的 D1-C，共六次运行，采用相同的审计。门控条件：寻找甜点区边界，并将第 5.1 节从一个点升级为一条曲线。
  2. 更强的逐字依赖探测 — 一项重读无法修复确切性损失（或使重读变得昂贵）的任务。门控条件：对调度策略的直接转换行提出任何主张。
  3. 第二个“轨迹 × 质量”数据点 — 在第 5.2 节的代理假设被用于任何地方之前，这是必须具备的前提。
  4. 受控微调 (NUDGE-GOV) — 安慰剂结果意味着需要一个极简的仪式性尾部；在未受控状态下，其成本达到了 6 倍。注入上限加上空悬挂停止机制。在分支 B 的后续跟进清理完毕前不予安排。
  5. 标定研究 (C‴) — 将“声明为可丢弃但被重读”作为每个模型的标定率进行测量；声明税数据（第 6.4 节）已掌握。这是论文轴线项：该比率是一个能力轮廓，可跨模型世代使用。
  6. 账本延续 (WS-6) — 从持久化账本中跨会话恢复工作语义状态。门控条件：狗食验证证据与保留规则。
  7. 服务商压缩分支 (E) — 当支持缓存观测的服务商公开服务器端压缩 API 时，将其作为分支进行对比，而不是围绕它重写论文。
  8. 调度策略硬化 — R2 映射（重构 → 投影；探索/完成度优先 → 检查点；探索/成本优先 → 原生或基线；直接转换 → 在当前探测强度下无特殊处理）仅作为推荐层发布，直至其在保留的任务家族中存活下来。

停止条件与队列一样是路线图的一部分：降低 Token 但提高产物失效的机制必须停止；仅在一种提示形状下胜出的机制保持实验性；使本地规则废弃的服务商特性将使该规则退役，并保留测量框架。

9. 关于本项目构建方式的说明

本仓库的代码是由 LLM 编码智能体根据自包含的任务数据包（规范、确切文件列表、验收标准、禁止区域）以测试驱动（test-first）方式编写的，这些数据包及其修订历史保留在仓库的目标账本中。人类的参与被刻意集中在决策层：参数选择、质量评审、门控判定以及可以提出哪些主张的决定——这些工作从未被委派。项目期间发生的两起运行事件（一次是压缩引起的上下文丢失影响了监管会话本身，另一次是较弱模型对硬性禁区的越界）均被捕获、回滚，并被编纂进协作协议中，而不是被丢弃。我们记录这一点，既是为了追溯源头，也是因为它是本文主题的一个微小实例：工作流本身就是一次决定信任哪些上下文、验证哪些上下文以及刻意遗忘哪些上下文的实践。

Deterministic Context Projection in an Agent Loop: An Auditable Account of the Compaction Tax

Working Paper · Draft v0.1 · 2026-07-07 Self-contained: all claims are backed by files in this repository. No external citations. Status: preliminary evidence (n=3 paired runs per cell). Open items are listed in §8.

智能体循环中的确定性上下文投影：压缩税的可审计账本

工作论文 · 草案 v0.1 · 2026-07-07 自包含：所有主张均由本仓库中的文件支持。无外部引用。 状态：初步证据（每组 n=3 次配对运行）。未决事项列于第 8 节。

Abstract

Coding agents accumulate conversation context faster than they consume it, and every serving stack now prices that accumulation in a specific way: input tokens that extend an unchanged prompt prefix are billed at roughly one tenth of the base rate, while any byte that breaks the prefix re-prices everything after it. Context compaction — replacing old conversation content with summaries — is therefore not merely a window-management technique; it is a transaction against a cache ledger, and the transaction has a price that the dominant implementation (LLM-generated summaries) makes impossible to audit.

This paper describes taucode (formerly ecode during the pi-experiment phase; name unified 2026-07 — τ = 2π), a small research harness built on the pi coding agent, that replaces LLM summarization with deterministic context projection: a pure function from message history to a smaller message history, applied only at send time, gated by a token threshold below which the payload is byte-identical to the unprojected one. Because the projection is deterministic and the raw history is never modified, every consequence — cache-hit dips and recoveries, forced re-reads, lost task state — becomes attributable and measurable. We use this property to decompose the compaction tax into three separately measurable terms (summary production cost, prefix-break cost, and semantic-loss cost), and to run paired four-arm experiments against a provider (DeepSeek) that exposes per-request cache telemetry.

The headline result is deliberately narrow: on refactor/code-production tasks, deterministic projection reached the same acceptance and equal-or-better human-reviewed code quality at roughly one third of the baseline cost (n=3, medians). The negative results are equally load-bearing: on exploration tasks the same mechanism failed to preserve task-progress semantics and produced the worst run in the dataset; a semantic-anchor repair was tested against a token-matched placebo and its completion claim did not survive. We report both, the audit machinery that caught two of our own wrong conclusions before publication, and a roadmap of gated follow-ups.

摘要

智能体在编码过程中积累对话上下文的速度远快于其消耗速度，而现在的每个推理服务栈都以特定方式对这种积累进行计费：扩展未改变前缀的输入 Token 按基础费率的约十分之一计费，而任何破坏前缀的字节都会重新计算其后所有内容的价格。因此，上下文压缩（用摘要代替旧的对话内容）不仅是一种窗口管理技术，而且是对缓存账本的交易，且该交易具有主导实现（LLM 生成的摘要）无法审计的价格。

本文介绍了 taucode（pi 实验阶段曾用名 ecode；2026-07 统一命名，取 τ = 2π），这是一个基于 pi 编码智能体构建的微型研究框架，它将 LLM 摘要替换为确定性上下文投影（deterministic context projection）：这是一个从消息历史到较短消息历史的纯函数，仅在发送时应用，并受 Token 阈值限制，低于该阈值时发送的负载与未投影的负载完全一致。由于投影是确定性的，且原始历史从未被修改，因此每个后果——缓存命中率的下降与恢复、强制重读、丢失的任务状态——都变得可追溯和可衡量。我们利用这一特性将压缩税分解为三个可单独衡量的项（摘要生成成本、前缀中断成本和语义损失成本），并针对提供单次请求缓存遥测的云服务商（DeepSeek）运行了配对的四臂实验。

核心结果故意收窄：在重构/代码生成任务中，确定性投影达到了相同的采纳率，并在人工评估中获得了相同或更好的代码质量，而成本仅为基线的三分之一左右（n=3，中位数）。负面结果同样具有支撑作用：在探索性任务中，相同的机制未能保留任务进度语义，并产生了数据集中最差的运行结果；在针对 Token 匹配的安慰剂对照测试中，语义锚点修复的完成声明未能通过验证。我们报告了这两者、在发表前捕捉到我们自身两个错误结论的审计机制，以及后续门控研究的路线图。

1. Introduction: the economics, from first principles

1.1 What an agent turn pays for

A coding agent is a loop: the harness sends the accumulated conversation (system prompt, user messages, assistant messages, tool calls, tool results) to a model, the model replies with text and/or tool calls, the harness executes the tools and appends the results, and the loop repeats. Three facts about this loop generate everything else in this paper.

Fact 1: context grows monotonically, and most of it is tool payload. Each turn appends the full text of files read, command output, and the full arguments of write/edit calls. In our transcripts, old tool payloads dominate context size within ten to twenty turns.

Fact 2: providers price prefix stability, not just size. Serving stacks reuse the attention KV state of a request prefix that is byte-identical to a previous request. Providers pass this saving on: cached input tokens are billed at roughly 0.1× the uncached rate (we use DeepSeek's pricing and its per-request telemetry fields, e.g. prompt_cache_hit_tokens, throughout). The corollary is strict: the cost of a turn is not a function of context length alone; it is a function of how many of those bytes extend an unchanged prefix. An agent that grows its context append-only pays the expensive rate once per byte and the cheap rate ever after.

Fact 3: any compaction is a prefix regime change. Rewriting history — however cleverly — changes bytes early in the prompt. Every subsequent token of the request is then uncached. Compaction therefore never "saves tokens" in a single step; it trades a one-time cache break for a shorter prefix whose future turns are cheaper. Whether the trade wins depends on how long the session continues after the break and on what the summary destroyed.

This yields the cost inequality that a compaction scheme must satisfy to be worth running (first stated in docs/reports/r2-turn-interaction-retro-2026-07-06.md):

saved payload cost  >  cache-discontinuity cost + extra-turn cost + re-read cost

The right-hand side is the compaction tax. We decompose it into three terms:

  • T1 — summary production cost: tokens spent generating the summary itself (an LLM summarizer pays output-token prices; a deterministic function pays zero);
  • T2 — prefix-break cost: the one-time loss of the cache discount on the rewritten span and everything after it;
  • T3 — semantic-loss cost: the downstream behavior induced by information the summary destroyed — re-reading files whose contents were summarized away, wandering trajectories, or failing to produce the required artifact at all.

T3 is the largest and least discussed term. It never appears in a token-bill comparison, because its worst case is not a bigger bill — it is a completed-looking session with a missing artifact.

1.2 Why the dominant implementation cannot be audited

The standard compaction mechanism is to ask a model to summarize the conversation when the window fills. This has two structural problems, independent of summary quality.

First, nondeterminism destroys attribution. If the same session state can produce different summaries on different runs, then any downstream failure — a re-read burst, a lost requirement — cannot be attributed to a specific, inspectable transformation. The experiment "did compaction cause this?" has no stable treatment.

Second, the summary rewrites history in place. Once the original messages are replaced in the session record, the counterfactual (what the model would have seen without compaction) is gone, and so is the ability to audit what was lost.

The design hypothesis of this project is that both problems are removable at once: make the transformation a pure, deterministic function applied only to the outgoing payload, leave the persisted history untouched, and the entire causal chain — which bytes changed, what the cache did in response, what the model did in response to that — becomes measurable from logs.

1.3 The claim, stated as an existence proposition

We do not claim deterministic projection is universally better. The program is an existence proposition with a boundary-mapping agenda:

∃ (workload, threshold):  task semantics preserved (acceptance not degraded)
                          ∧ net cost < append-only baseline

Section 5 exhibits one proven point (refactor workload × 32k-token threshold). Section 6 maps where the proposition fails and why. Section 8 lists the sweeps that turn the point into a region.

1. 引言：基于第一性原理的经济学

1.1 智能体单次轮次支付的费用

编码智能体是一个循环：框架将累积的对话（系统提示词、用户消息、助手消息、工具调用、工具结果）发送给模型，模型回复文本和/或工具调用，框架执行工具并追加结果，循环往复。关于这个循环的三个事实引出了本文的其余所有内容。

事实 1：上下文呈单调增长，且大部分是工具载荷。 每一轮都会追加读取文件的全文、命令输出以及写入/编辑调用的完整参数。在我们的转录本中，旧的工具载荷在十到二十轮之内就占据了上下文的主要大小。

事实 2：服务商对前缀稳定性进行定价，而不仅仅是大小。 服务栈会重用与先前请求字节完全相同的请求前缀的 attention KV 状态。服务商将这种节省回馈给用户：缓存的输入 Token 按未缓存费率的约 0.1 倍计费（我们全程使用 DeepSeek 的定价及其单次请求遥测字段，例如 prompt_cache_hit_tokens）。其推论是严苛的：单次轮次的成本不仅是上下文长度的函数，也是这些字节中有多少扩展了未改变前缀的函数。 一个采用仅追加方式增长上下文的智能体，对每个字节仅支付一次昂贵的费率，之后便一直享受便宜的费率。

事实 3：任何压缩都是前缀规则的改变。 重写历史——无论多么聪明——都会改变 Prompt 早期的字节。随后请求中的每个 Token 都会变得未缓存。因此，压缩绝不会在单一步骤中“节省 Token”；它是用一次性的缓存中断换取更短的前缀，从而使未来的轮次更便宜。这种交易是否划算，取决于中断后会话持续了多久，以及摘要破坏了什么内容。

这得出了一个压缩方案必须满足的成本不等式，才值得运行（首次阐述于 docs/reports/r2-turn-interaction-retro-2026-07-06.md）：

节省的载荷成本  >  缓存不连续性成本 + 额外轮次成本 + 重读成本

等式的右侧即为压缩税。我们将其分解为三项：

  • T1 — 摘要生成成本：生成摘要本身所消耗的 Token（LLM 摘要器需要支付输出 Token 的价格；确定性函数则支付零成本）；
  • T2 — 前缀中断成本：重写区间及其后所有内容一次性失去缓存折扣的损失；
  • T3 — 语义损失成本：摘要破坏的信息所引起的下游行为——重新读取已被摘要过滤掉的文件内容、迷失的轨迹，或者根本无法生成所需的产物。

T3 是最大且最少被讨论的项。它从未出现在 Token 账单对比中，因为它的最坏情况不是更大的账单——而是一个看起来已完成但缺失了产物的会话。

1.2 为什么主导实现无法被审计

标准的压缩机制是在窗口填满时让模型总结对话。无论摘要质量如何，这都存在两个结构性问题。

首先，非确定性破坏了归因。如果相同的会话状态在不同的运行中会产生不同的摘要，那么任何下游失败——重读爆发、遗漏需求——都无法归因于特定的、可检查的转换。对于“压缩是否导致了这一结果？”这一实验，没有稳定的处理对照。

其次，摘要会就地重写历史。一旦会话记录中的原始消息被替换，反事实（如果模型在没有压缩的情况下会看到什么）就消失了，审计丢失内容的能力也随之消失。

本项目的核心设计假设是，这两个问题可以同时消除：将转换设计为仅应用于发送载荷的纯确定性函数，保持持久化的历史记录不受影响，这样整个因果链——哪些字节发生了变化、缓存做出了何种响应、模型又对此做出了何种反应——都可以从日志中进行衡量。

1.3 声明：作为存在性命题阐述

我们并不声称确定性投影普遍更好。该程序是一个具有边界映射议程的存在性命题：

∃ (工作负载, 阈值):  任务语义得以保留 (采纳率未降低)
                          ∧ 净成本 < 仅追加基线

第 5 节展示了一个已证实的点（重构工作负载 × 32k Token 阈值）。第 6 节映射了该命题在何处失败以及原因。第 8 节列出了将点扩展为区域的参数扫频。

2. Design principles

Four invariants govern the implementation. Each is enforced by tests, not by convention.

P1 — Determinism: same state, same bytes. The projection is a pure function of the message array and its configuration. Given the same input it renders the same output, byte for byte. This is what makes the projected prefix cacheable again after the one-time break (T2 is paid once, not per turn), and what makes failures attributable.

P2 — Identity below threshold. Below a configured token estimate (compactAfterInputTokens, default 32,000) the projection returns the same array reference, unchanged. Two consequences: sessions that never grow large pay zero — the mechanism can be permanently installed without a tax on small sessions; and the prefix of a warm session is never perturbed early, when cache value is highest. The same discipline applies within a projected send: if no message qualified for compaction, the identity is returned rather than a new-but-equal array (extensions/deterministic-compaction/src/projection.ts:99-103).

P3 — Raw history is append-only; projection exists only at send time. The transformation is applied in the harness's context hook, which shapes the outgoing request payload and is never persisted (verified against the host harness in docs/g0-survey.md, Item 3). The session file always contains the full, unmodified history. Auditing "what was hidden from the model at turn N" is a replay of a pure function, not an archaeology of overwritten records.

P4 — Provenance by content hash. Summaries and hints never assert facts about file state in prose; they carry content hashes and diffstats computed at tool-execution time. Anything the model is told about the past is mechanically checkable against the workspace.

Two further disciplines follow from P1–P3 and are worth naming because they exclude a known failure class. Idempotency: a projected summary is smaller than every eligibility threshold, so projecting twice is a no-op — the hook can run on every send without compounding. No silent prefix perturbation: every volatile addition (trust hints, anchors; §3.4–3.5) is appended strictly at the tail of the outgoing payload, after the stable prefix, so the cached span is never broken by advisory content.

2. 设计原则

四个不变量支配着实现。每一个都由测试强制执行，而非约定俗成。

P1 — 确定性：状态相同，字节相同。 投影是消息数组及其配置的纯函数。给定相同的输入，它会渲染出字节完全相同的输出。这就是为什么被投影的前缀在一次性中断后可以再次被缓存（T2 只支付一次，而不是每轮都付），也是为什么失败是可以归因的原因。

P2 — 低于阈值时保持原样。 在配置的 Token 估算值（compactAfterInputTokens，默认 32,000）以下，投影返回相同的数组引用而不做任何修改。两个后果：从未变大的会话支付零成本——该机制可以永久安装，而不会对小型会话征税；并且在会话发热的早期（此时缓存价值最高），前缀绝不会受到扰动。在投影发送内部也应用相同的纪律：如果没有消息符合压缩条件，则返回恒等引用，而不是一个相同但新建的数组（extensions/deterministic-compaction/src/projection.ts:99-103）。

P3 — 原始历史仅追加；投影仅在发送时存在。 转换在框架的 context 钩子中应用，该钩子塑造了发送的请求载荷，并且从未被持久化（针对宿主框架的验证见 docs/g0-survey.md 第 3 项）。会话文件始终包含完整的、未经修改的历史记录。审计“在第 N 轮对模型隐藏了什么”是对纯函数的重放，而不是对被重写记录的考古。

P4 — 通过内容哈希追溯源头。 摘要和提示绝不在散文中声称关于文件状态的事实；它们携带在工具执行时计算的内容哈希和 diff 统计信息。模型被告知的关于过去的任何事情，都可以在工作区中进行机械式验证。

另外两个准则源自 P1–P3，值得提及，因为它们排除了一类已知的失败模式。幂等性：投影摘要小于每个符合条件的阈值，因此投影两次是无操作（no-op）——钩子可以在每次发送时运行而不会产生累积效应。无静默前缀扰动：每个易失性添加（信任提示、锚点；第 3.4–3.5 节）都严格追加在发送载荷的末尾，即在稳定前缀之后，因此缓存的区间绝不会被建议性内容破坏。

3. System

3.1 Host: a thin loop with explicit seams

taucode is built on pi (the pi-mono coding agent), chosen precisely because it is a thin loop: its context assembly is small enough to be fully observable, and it exposes two extension seams that map exactly onto the design:

  • Seam A — a context hook that runs before every LLM call and may transform the outgoing message array; its return affects only the send payload and is never persisted.
  • Seam B — a session_before_compact hook that may replace pi's native LLM-summary compaction with a caller-supplied entry (kept optional and off by default in taucode; the native summarizer is retained as an experiment arm, not deleted).

The fork discipline is strict: pi/ carries zero diff against upstream; everything lives in an out-of-tree extension (extensions/deterministic-compaction/) and a standalone package (packages/compaction-core/). The extension is loaded by path (pi --extension .../src/extension.ts) and carries its mechanisms behind environment flags, all default-off except the projection itself.

3.2 The projection core: messages → messages

packages/compaction-core/ (~1,400 lines, no dependency on any harness code) implements the pure function. Its entry point:

compactCodeProductions(
  messages: Message[],
  options?: Partial<CompactionOptions>,   // keepRecentAssistantMessages=3,
                                          // minArgTokens=800, minResultTokens=200
  injection?: CompactionInjection,        // five seams, all defaulting to
): CompactionResult                       // original behavior

The function walks the message array once (O(n)), pairs tool calls with their results, and replaces two kinds of payload with typed, deterministic summaries:

  • Old write/edit tool-call arguments become a CodeProductionSummary: {path, chars, lines, head, tail, result, hash?, diffstat?} — the head/tail preserve a five-line preview; the hash and diffstat (when a ledger is injected, §3.4) give the summary provenance.
  • Old read/bash/search/find results become per-tool summaries that keep the locator and drop the content: a read summary keeps {path, lines, chars, hash?} plus the fixed instruction "Re-run read if exact content is needed"; bash keeps {command, exitCode, stderrLines}; search keeps {pattern, matchCount, fileCount, preview}.

Protection rules bound the blast radius: the last keepRecentAssistantMessages assistant turns are never touched; error results are never compacted; small payloads below the token floors are never compacted; a summary that would not be strictly smaller than its source is discarded. Every replacement is recorded in a diffs array ({turn, toolName, path, rawTokens, compactedTokens, tokensSaved}), which is the observability substrate for everything in §4.

Five injection seams (strategy set, tool-name matcher, path/hash extractor, per-path summary metadata, protected paths) let a harness adapt the core without forking it; every seam defaults to the original behavior, and the no-injection output is pinned byte-exact by tests.

3.3 Hybrid gating at the send boundary

The extension wires the core into Seam A (extensions/deterministic-compaction/src/projection.ts):

function projectContext(messages, config): ProjectionOutcome {
  const rawTokens = estimateAgentTokens(messages);      // pi's own estimator
  if (rawTokens < config.compactAfterInputTokens) {
    return { messages, projected: false, rawTokens };   // identity (P2)
  }
  const compaction = compactCodeProductions(...);
  if (compaction.compactedCount === 0) {
    return { messages, projected: false, rawTokens };   // identity again
  }
  return { messages: projected, projected: true, rawTokens, compaction };
}

The gate deliberately reuses the host's token estimator rather than a bespoke one: the trigger only needs to be consistent, while cost accounting uses provider- reported usage (a lesson recorded as a metric-unit failure class, §6.4). Below the gate, the send payload is byte-identical and the provider's prefix cache absorbs the whole history at the discounted rate. Above it, the projection produces a new — but deterministic, hence re-cacheable — prefix. This is the mechanism behind the transition shape measured in §5.3: one dip, then recovery.

3.4 The trust protocol: hashes against confabulation

Compacting a read result creates a hazard: the model may "remember" file content that has since changed, and a summary that merely says "you read this file" does not tell it whether that memory is current. The trust protocol makes staleness mechanical.

A session-scoped ledger records, per path, the content hash and diffstat of the latest read/edit (trust-ledger.ts). At send time, staleViewHints (trust-hint.ts:30) scans the outgoing messages for read views whose content hash differs from the ledger's current hash for paths the model itself has since edited, and emits at most one line per path:

[stale-view] src/config.ts: view from a1b2c3d4 predates your edit at turn 12
(now e5f6a7b8); re-read only if you need current content.

The hint is appended as a volatile tail message — after the stable prefix, never persisted, replaced (not accumulated) each turn. The wording is deliberately non-prescriptive ("re-read only if you need"): the protocol supplies verifiable temporal facts and leaves the decision to the model.

Field validation (§5.4) shows a production-tier model using these hints exactly as designed, including the hard case: distinguishing "stale because I edited it" (hash mismatch, re-read) from "current because I was the last editor" (hash match, no re-read).

3.5 Semantic anchors: the same channel, for task state

The exploration failure mode (§6.1) motivated a second tail block: a deterministic, LLM-free digest of work state — files read (path + hash + turn), edits made (path + diffstat + hash, failures recorded as failures), tests run (command + parsed result), and pending acceptance targets — accumulated from tool events by a WorkAnchor (anchor.ts), rendered only on projected turns. Its discipline mirrors the trust protocol: every line is a mechanically verifiable fact; no evaluative language ("almost done") is permitted. Whether this block buys anything is an empirical question, and §6.2 reports the answer we got, including the placebo control that dismantled our first interpretation.

3.6 Observability

Every mechanism reports into the same surfaces: per-turn JSONL rows (projected flag, per-path compaction diffs, provider usage including cache-hit tokens and reasoning tokens, re-read counters), a TUI gate widget showing threshold distance and trigger markers, a cache-hit trace, and ambient telemetry rows for ordinary (non-experiment) sessions. Machine-readable tail evidence blocks record exactly which volatile block (anchor, placebo, nudge, substitution) was present on which turn, so post-run audit can distinguish "tail rendered but ignored" from "tail absent" (tail-evidence.ts; requirement OBS-TAIL, §6.2). The extension carries 137 unit tests; the projection's identity and byte-stability properties are pinned by fixtures.

3. 系统

3.1 宿主：具有显式接缝的微型循环

taucode 构建在 pi（pi-mono 编码智能体）之上，选择它正是因为它是一个微型循环：其上下文组装足够小，完全可观测，并且它暴露了两个与设计完全契合的扩展接缝：

  • 接缝 A — 一个在每次 LLM 调用之前运行的 context 钩子，可以转换 outgoing 消息数组；其返回值仅影响发送载荷，绝不持久化。
  • 接缝 B — 一个 session_before_compact 钩子，可以用调用者提供的条目替换 pi 原生的 LLM 摘要压缩（在 taucode 中保持可选且默认关闭；原生摘要器作为实验分支保留，未被删除）。

分支规范非常严格：pi/ 与上游相比没有任何 diff；所有内容都存在于树外扩展（extensions/deterministic-compaction/）和独立包（packages/compaction-core/）中。扩展通过路径加载（pi --extension .../src/extension.ts），并将其机制置于环境变量标志之后，除投影本身外，所有标志默认关闭。

3.2 投影核心：messages → messages

packages/compaction-core/（约 1,400 行，不依赖于任何框架代码）实现了该纯函数。其入口点为：

compactCodeProductions(
  messages: Message[],
  options?: Partial<CompactionOptions>,   // keepRecentAssistantMessages=3,
                                          // minArgTokens=800, minResultTokens=200
  injection?: CompactionInjection,        // 五个接缝，全部默认为原始行为
): CompactionResult

该函数遍历消息数组一次（O(n)），将工具调用与其结果配对，并用类型化的、确定性的摘要替换两种载荷：

  • 旧的写入/编辑工具调用参数变为 CodeProductionSummary： {path, chars, lines, head, tail, result, hash?, diffstat?} —— head/tail 保留了五行预览；哈希和 diffstat（当注入账本时，第 3.4 节）为摘要提供了来源。
  • 旧的读取/bash/搜索/查找结果变为每个工具的摘要，保留定位符并丢弃内容：read 摘要保留 {path, lines, chars, hash?} 以及固定指令“如果需要确切内容，请重新运行 read”；bash 保留 {command, exitCode, stderrLines}；search 保留 {pattern, matchCount, fileCount, preview}。

保护规则限制了影响范围：最后 keepRecentAssistantMessages 轮助手回合绝不会被触及；错误结果绝不压缩；低于 Token 下限的小载荷绝不压缩；不会比其源内容更小的摘要将被丢弃。每次替换都会记录在 diffs 数组中（{turn, toolName, path, rawTokens, compactedTokens, tokensSaved}），这是第 4 节中所有内容的可观测性基质。

通过五个注入接缝（策略集、工具名称匹配器、路径/哈希提取器、单路径摘要元数据、受保护路径），框架可以在不分叉核心的情况下对其进行适配；每个接缝默认使用原始行为，且无注入输出通过测试进行字节精确的固定。

3.3 发送边界处的混合门控

扩展将核心连接到接缝 A（extensions/deterministic-compaction/src/projection.ts）：

function projectContext(messages, config): ProjectionOutcome {
  const rawTokens = estimateAgentTokens(messages);      // pi 自己的估算器
  if (rawTokens < config.compactAfterInputTokens) {
    return { messages, projected: false, rawTokens };   // 恒等 (P2)
  }
  const compaction = compactCodeProductions(...);
  if (compaction.compactedCount === 0) {
    return { messages, projected: false, rawTokens };   // 再次恒等
  }
  return { messages: projected, projected: true, rawTokens, compaction };
}

该门控故意重用了宿主的 Token 估算器，而不是定制的估算器：触发器只需要保持一致即可，而成本核算使用服务商报告的用量（这是从度量单位失效类别中汲取的教训，第 6.4 节）。在门控之下，发送载荷完全一致，服务商的前缀缓存以折扣费率吸收整个历史记录。在门控之上，投影产生一个新的——但由于是确定性的，因此可以重新缓存——前缀。这就是第 5.3 节中测得的过渡形状背后的机制：一次下降，然后恢复。

3.4 信任协议：用哈希对抗胡言乱语

压缩 read 结果会带来风险：模型可能会“记住”此后已更改的文件内容，而仅显示“您读取了此文件”的摘要无法告诉它该记忆是否为最新。信任协议使陈旧性判断变得机械化。

会话范围的账本记录了每个路径最新读取/编辑的内容哈希和 diff 统计信息（trust-ledger.ts）。在发送时，staleViewHints（trust-hint.ts:30）扫描 outgoing 消息，寻找其内容哈希与账本当前哈希不同的 read 视图（针对模型自身此后编辑过的路径），并为每个路径最多输出一行提示：

[stale-view] src/config.ts: view from a1b2c3d4 predates your edit at turn 12
(now e5f6a7b8); re-read only if you need current content.

提示作为易失性尾部消息追加——在稳定前缀之后，绝不持久化，每轮都会被替换（而不是累加）。其措辞故意是非指令性的（“仅在需要时重读”）：该协议提供可验证的临时事实，并将决定权留给模型。

现场验证（第 5.4 节）显示，生产级模型完全按照设计使用这些提示，包括困难情况：区分“因我编辑而陈旧”（哈希不匹配，重读）与“因我是最后的编辑者而最新”（哈希匹配，不重读）。

3.5 语义锚点：相同通道中的任务状态

探索失败模式（第 6.1 节）促成了第二个尾部块的产生：一个确定性的、无 LLM 参与的工作状态摘要——读取的文件（路径 + 哈希 + 轮次）、做出的编辑（路径 + diffstat + 哈希，失败记录为失败）、运行的测试（命令 + 解析后的结果）以及悬而未决的验收目标——这些信息由 WorkAnchor（anchor.ts）从工具事件中收集，仅在投影轮次中渲染。其规范镜像了信任协议：每一行都是机械可验证的事实；不允许使用任何评估性语言（例如“即将完成”）。这个块是否能带来收益是一个实证问题，第 6.2 节报告了我们得到的答案，包括解构了我们最初解释的安慰剂对照组。

3.6 可观测性

每个机制都向相同的界面报告：逐轮 JSONL 行（投影标志、单路径压缩 diff、服务商用量（包括缓存命中 Token 和推理 Token）、重读计数器）、一个显示阈值距离和触发标记的 TUI 门控挂件、一个缓存命中追踪，以及用于普通（非实验）会话的环境遥测行。机器可读的尾部证据块精确记录了哪一轮存在哪个易失性块（锚点、安慰剂、微调、替换），以便在运行后审计时能够区分“尾部已渲染但被忽略”与“尾部缺失”（tail-evidence.ts；要求 OBS-TAIL，第 6.2 节）。扩展包含 137 个单元测试；投影的恒等和字节稳定性属性已通过固定装置锁死。

4. Measurement methodology

The harness exists to make one kind of sentence possible: "arm X differed from arm Y by Z, and here is the artifact trail." This section describes the controls that license such sentences at all.

4.1 Paired arms on frozen workspaces

The unit of measurement is a packet: a task prompt plus a workspace snapshot plus mechanical acceptance criteria. Three packets span the hypothesized regimes:

  • R1 — refactor/code-production: read large sources once, produce new modules;
  • E1 — exploration: survey a codebase and produce a required artifact (SUBSYSTEM-MAP.md) plus answers;
  • D1 — direct transform: convert data where exact original text may be needed after compaction (a designed negative probe).

Each packet runs under four arms: A (baseline: no compaction of any kind), B′ (pi's native LLM-summary compaction at a comparable window), C (deterministic projection, Seam A), D (hybrid: projection plus Seam B checkpoint). Workspaces are materialized from snapshots whose manifest hash is verified per run — all arms start from byte-identical trees. Arm order is re-randomized per repeat and interleaved across packets to prevent provider time-of-day effects from aliasing into arm effects. Model, provider, thresholds, and flags are recorded in a per-run manifest; n=3 repeats per cell.

Cost is computed from provider-reported usage in the unit uncached_input + 0.1 × cached_input (thousands), reflecting the actual pricing asymmetry; medians are reported with full lists alongside.

4.2 Codified judgment gates

Two failure classes are checked by pure functions over run summaries (experiments/lib/gates.ts), not by reviewer discretion:

  • invalid — the arm's mechanism never engaged (a projection arm whose threshold was never crossed; a native arm where the summarizer never fired). Deltas against such runs are meaningless and are excluded before interpretation.
  • suspicious — false savings: total tokens went down relative to baseline while churn went up (re-read count or compacted-path re-read rate increased). This encodes the project's founding negative result — token savings that are actually the model re-buying destroyed information — as a mechanical flag.

Both gates return the numbers that fired them and refuse to editorialize. Their fixture tests are part of the repository's test suite.

4.3 Controls above the gates

Three further disciplines proved decisive. Placebo tails: when testing whether an informative tail block (the anchor) helps, a token-matched but content-free tail is run as its own arm — pre-registered before the comparison, not improvised after (placebo-tail.ts). Blind artifact review: acceptance scripts bound quality from below; a human review of produced code, blinded to arm labels, is the actual quality instrument (§5.2). Verdict separation: run reports are evidence; a single verdict document (docs/stage-verdict-2026-07.md) holds the claims and their allowed phrasings, and downgrades every report to an appendix. Nothing is citable twice.

4.4 What the discipline caught

The methodology section would be decoration if it had never fired. It fired twice.

Round 1 produced a striking "projection causes 2.5× turn inflation" finding. The gates flagged half of round 1 as invalid (mechanisms not engaged), the finding was held out of the verdict, and round 2 (n=3) reversed it outright: the same arm ran at median 18 turns versus baseline 36. The retraction is recorded in docs/reports/r2-verdict.md (ruling 1).

Later, the semantic anchor's completion effect (§6.2) was pre-registered against a placebo; the placebo matched it, and the intended headline claim was withdrawn before it was ever made public. Both events are treated as products of the harness, not embarrassments to it: the audit trail is the deliverable.

4. 测量方法论

该测试框架的存在是为了让以下陈述成为可能：“分支 X 与分支 Y 在 Z 上的差异如下，这是对应的产物轨迹。” 本节描述了使此类陈述具有可信度的控制措施。

4.1 冻结工作区上的配对分支

测量的单位是数据包 (packet)：任务提示词加上工作区快照，再加上机械化验收标准。三个数据包涵盖了假设的运作状态：

  • R1 — 重构/代码生成：一次性读取大型源文件，生成新模块；
  • E1 — 探索：调查代码库并生成所需的产物（SUBSYSTEM-MAP.md）以及解答；
  • D1 — 直接转换：转换数据，在压缩后可能需要确切的原始文本（设计好的负面探测）。

每个数据包在四个分支下运行：A（基线：不进行任何形式的压缩），B′（pi 在同等窗口下的原生 LLM 摘要压缩），C（确定性投影，接缝 A），D（混合型：投影加上接缝 B 检查点）。工作区是从清单哈希经过逐次运行验证的快照中实例化的——所有分支都从字节相同的目录树开始。分支的顺序在每次重复时重新随机化，并在数据包之间交错进行，以防止服务商在一天中不同时间的性能波动影响分支效果。模型、服务商、阈值和标志都记录在每次运行的清单中；每个单元重复 n=3 次。

成本根据服务商报告的用量计算，单位为 uncached_input + 0.1 × cached_input（千个 Token），这反映了实际的计费不对称性；中位数与完整列表并排报告。

4.2 程式化的评估门控

运行摘要（experiments/lib/gates.ts）上的两个失效类别由纯函数检查，而非评审员酌情决定：

  • invalid — 分支的机制从未生效（阈值从未被跨越的投影分支；摘要器从未启动的原生分支）。这些运行的差异是没有意义的，并在解释之前被排除。
  • suspicious — 虚假节省：相对于基线，总 Token 数下降，但流失率上升（重读计数或压缩路径重读率增加）。这以机械标志的形式编码了项目创立之初的负面结果——即模型通过重新获取已被破坏的信息来换取 Token 的节省。

两个门控都只返回触发它们的数据，拒绝进行主观评论。它们的固定装置测试是仓库测试套件的一部分。

4.3 门控之上的控制

另外三项纪律被证明是决定性的。安慰剂尾部：在测试信息性尾部块（锚点）是否有帮助时，会运行一个 Token 大小匹配但内容为空的尾部作为其自身的分支——在对比前预先注册，而不是事后临时起意（placebo-tail.ts）。双盲产物评审：验收脚本从下方约束质量；对生成的代码进行不带分支标签的人工评审，是实际的质量评估工具（第 5.2 节）。结论分离：运行报告是证据；一份单独的结论文档（docs/stage-verdict-2026-07.md）保存了主张及其允许的措辞，并将每份报告降级为附录。任何内容都不能被引用两次。

4.4 规范所捕获的问题

如果这些规范从未发挥作用，那方法论章节就成了摆设。事实上，它发挥了两次作用。

第一轮实验产生了一个引人注目的结论：“投影导致了 2.5 倍的轮次膨胀”。然而，评估门控将第一轮的一半运行标记为无效（机制未生效），该发现被排除在结论之外，而第二轮（n=3）则完全逆转了这一结果：相同的分支以中位数 18 轮运行，而基线为 36 轮。这一撤回记录在 docs/reports/r2-verdict.md 中（裁决 1）。

后来，语义锚点的完成效果（第 6.2 节）针对安慰剂组进行了预注册；结果安慰剂组与它打平，预期的核心主张在公之于众之前就被撤回了。这两起事件都被视为该框架的产物，而不是它的尴尬：审计轨迹本身就是交付物。

5. Results

All numbers below are from paired real-provider runs (DeepSeek v4-flash via pi), 27 core runs plus 9 add-on runs, n=3 per cell, medians with full lists in the source reports (docs/reports/r2-verdict.md, r2-turn-interaction-retro-2026-07-06.md). Claims are stated with the boundary they were measured under and none beyond it.

5.1 The proven point: refactor × deterministic projection

Packet R1 (refactor) A (baseline) B′ (native) C (projection) D (hybrid)
cost, median [list] 1342 [492, 1342, 1744] 601 [433, 601, 762] 449 [401, 449, 857] 879

C's median cost is 33% of baseline with acceptance intact across all repeats, and — the mechanistically decisive column — zero re-reads of compacted paths: nothing the projection destroyed was ever needed again. The first-principles explanation is that refactor work converts read payloads into workspace edits; after the conversion, the durable state is the workspace, and the old bytes in context are pure debt. This is the workload the mechanism was designed for, and it behaves as designed.

5.2 Quality, bounded from above the scripts

Static checks bound quality from below. A blinded per-file human review of all three R1-C runs (review-R1-C-code-quality.md, summarized in the R2 verdict addendum) upgraded the claim: the cheapest run (365k tokens, 17 turns) was also judged the best code (clean encapsulation, no TODOs; sole blemish an export *), and the most expensive run was judged the worst. The cost claim is therefore cost-and-quality confirmed at n=3, not cost-at-quality's-expense.

The review produced a side finding we now treat as the first data point of a hypothesis: trajectory wandering and code degradation moved together — the worst-quality run was also the most expensive one. If trajectory length is a usable quality proxy, cost and quality are not a trade-off frontier at this scale but two projections of the same variable (whether the model stayed oriented). This is promoted to a measurement target, not asserted (§8).

5.3 The cache transition curve

Because the provider reports per-request cache hits, the T2 term is directly observable. The canonical good shape, from run 19-r3-G2-R1-C.jsonl:

turn payload input tokens cache-read tokens
10 raw 28,042 30,336
11 first projected 9,620 2,304
13 projected 14,825 13,312
15 projected 15,412 13,312
17 projected 16,109 20,608

One dip at the regime change, then recovery as the deterministic rendering holds byte-stable — exactly the trade described in §1.1: a one-time break purchasing a shorter, once-again-cached prefix. The counter-example run 14-r2-G2-E1-C shows why this curve is subordinate to semantics: its cache also recovered, and the run still produced no artifact. A cheap stable prefix is worthless to a model that no longer knows what it was doing — which is the bridge to §6.

5.4 The trust protocol in the field

A live flag-on session (docs/reports/v2-tp-wild-field-run.md) validated the stale-view mechanism end to end: no false positives when views were current; hint fired on a genuinely stale read and the model re-read before answering (correctly); and in the multi-file stress round, the model re-read only files where its view hash mismatched the ledger, declining to re-read files where it was itself the last editor. Cache-hit rate was unaffected by the tail append (the observed drop was attributable to re-read tokens, not prefix breakage). The protocol's design bet — that mechanical temporal facts, not prose confidence, are what a model can act on correctly — held under production conditions.

5.5 Self-application

The blind review in §5.2 was itself conducted inside a taucode session: the audit work ran with projection live, saving 56,780 tokens (67%) at a 98.1% cache-hit rate. We flag this as dogfooding, not as controlled evidence; its value is that the tooling's own paper trail was produced under the tooling.

分支 C 的中位数成本是基线的 33%，且在所有重复实验中验收率完好无损，并且——在机制上起决定性作用的指标——压缩路径的重读数为零：投影破坏的任何内容都未被再次使用。从第一性原理来看，解释是重构工作将读取的载荷转化为工作区编辑；在转换之后，持久状态就是工作区，而上下文中旧的字节纯粹是债务。这就是该机制所针对的工作负载，其表现也符合预期。

5.2 高于脚本约束的质量评估

静态检查从下方约束了质量。对所有三个 R1-C 运行进行不带标签的逐文件人工评审（review-R1-C-code-quality.md，总结在 R2 结论增补中）提升了这一主张：成本最低的运行（365k Token，17 轮）也被评为质量最佳的代码（干净的封装，无 TODO；唯一的瑕疵是 export *），而最昂贵的运行被评为最差的代码。因此，在 n=3 的水平上，成本主张被证实为“低成本兼具高质量”，而不是以牺牲质量为代价换取低成本。

评审产生了一个副发现，我们现在将其视为一个假设的第一个数据点：轨迹漂移与代码降级是同向运动的——质量最差的运行同时也是成本最高的运行。如果轨迹长度是一个可用的质量代理指标，那么在这种规模下，成本和质量就不是此消彼长的关系，而是同一变量的两个投影（即模型是否保持了方向感）。这被提升为一个衡量目标，而非既成断言（第 8 节）。

5.3 缓存过渡曲线

由于服务商报告了单次请求的缓存命中情况，因此 T2 项是直接可观测的。来自运行 19-r3-G2-R1-C.jsonl 的典型良好形态如下：

轮次 载荷 输入 Token 缓存读取 Token
10 原始 (raw) 28,042 30,336
11 首次投影 9,620 2,304
13 投影 14,825 13,312
15 投影 15,412 13,312
17 投影 16,109 20,608

在规则改变时出现一次下降，然后随着确定性渲染保持字节稳定而恢复——这正是第 1.1 节中描述的交易：一次性中断换取更短、再次被缓存的前缀。反例运行 14-r2-G2-E1-C 展示了为什么这条曲线从属于语义：它的缓存虽然也恢复了，但该运行依然未能生成任何产物。对于一个不再知道自己在做什么的模型来说，一个便宜且稳定的前缀毫无价值——这正是通往第 6 节的桥梁。

5.4 现场的信任协议

一次开启标志的实地运行会话（docs/reports/v2-tp-wild-field-run.md）端到端验证了陈旧视图机制：当视图为最新时无误报；在一次真正陈旧的读取上触发了提示，模型在回答前重新读取了内容（表现正确）；并且在多文件压力测试轮中，模型仅重新读取了其视图哈希与账本不匹配的文件，而没有重新读取它本身作为最后编辑者的文件。缓存命中率未受尾部追加的影响（测得的下降可归因于重读 Token，而非前缀破坏）。该协议的设计赌注——模型能够正确应对的是机械的临时事实，而非散文式的确定性陈述——在生产条件下得到了证实。

5.5 自主应用

第 5.2 节中的双盲评审本身就是在 taucode 会话中进行的：审计工作在投影启用的情况下运行，在 98.1% 的缓存命中率下节省了 56,780 个 Token（67%）。我们将其归为“吃自家狗粮”（dogfooding），而不是受控的实验证据；它的价值在于，该工具自身的纸面记录就是在此工具下生成的。

compacted-path re-read rate 0 0 .231, .211, .625 .150, .333, .174

On exploration, projection's cost advantage vanishes (1616 vs 1527) and its completion is unstable. The hard failure (14-r2-G2-E1-C): twelve projected turns, zero artifact files, the required SUBSYSTEM-MAP.md never produced, re-read bursts concentrated on compacted paths. The mechanism is legible in the logs: projection preserved file evidence (paths, hashes, previews) but destroyed progress semantics — what has been mapped, what remains, what artifact is owed. When the model lost orientation it re-opened compacted files (the re-read rate column is this signal) or, in the worst case, read and reasoned in circles without ever crossing the artifact-production boundary.

Honesty requires the baseline column too: A also failed once (0/5). Losing the thread on exploration tasks is partly task nature, not purely a compaction pathology. But the paired shape stands: what exploration needs preserved is task state, and extractive summaries do not carry it. Meanwhile arm D completed 6/6 across two rounds (the dataset's only such cell) at ~1.6× baseline cost — its checkpoint keeps a persistent progress digest in context. The verdict phrasing is deliberate: hybrid+checkpoint buys completion on exploration, not cost.

6.2 The anchor's completion claim died by placebo

The semantic anchor (§3.5) was built as the direct, LLM-free response to F1: if the missing ingredient is progress state, restate it deterministically on every projected turn. First results looked confirmatory. The controlled comparison (Branch B, with OBS-TAIL evidence on every turn) says otherwise:

E1 arm (n=3) completion median turns median total tokens
C″ (anchor) 2/3 21 677,535
C-SB (anchor + sideband summaries) 1/3 21 709,726
C+PL (token-matched placebo tail) 3/3 28 882,010

A content-free tail block of matched size completed the task more reliably than the informative anchor. The ruling (R14 in the design log) splits the variable the first interpretation had conflated: the ritual buys completion; the content buys trajectory. Any same-shaped reminder at projected turns appears to re-orient the model toward finishing; the anchor's verifiable facts show up instead in the cost column (C″ was the cheapest E1 arm observed, one run at 19 turns / 538k) — shorter, less wandering trajectories, not higher completion. Two corollaries went into the roadmap rather than the narrative: a governed minimal nudge (the ungoverned variant completed 3/3 but at ~6× cost — 84 turns, 3.23M tokens — pure ritual with no content is an expensive superstition), and a calibration study, since "declared disposable but re-read" is now measurable per path.

6.3 F2 — churn hidden by passing checks: the negative probe that didn't bite

D1, the designed negative zone (exact-text dependency under an aggressive 8k threshold), failed to produce the expected failure: C passed 4/4 ×3 with median cost below baseline. But the logs show why it survived, and the why is a warning: compacted-path re-read rates ran as high as .917 and .926 — the model repeatedly asked for exact content, the harness repeatedly compressed it away, and re-reading repaired the loss each time. The probe was survivable by paying T3 in small installments. We record this as "the current probe is too weak," not "the negative zone is empty"; a stronger verbatim-dependency probe is queued (§8). The general lesson is methodological: acceptance scripts alone would have called this a clean pass; the churn counters are what kept it honest.

6.4 Smaller instruments that failed and were kept

Sideband summaries (C-SB) — off-thread LLM summarization of tool results — violated their own economics in the current implementation: one run spent 186 sideband calls / 158k output tokens to complete 0/5. The net-flow accounting (summaries must cost less than they save) is now a mock-level precondition for any rerun; the design is unproven, not disproven, and stays parked. Native-trigger nondeterminism (F3): pi's window-pressure compaction fired only 1/3 on the small packet, invalidating two B′ runs — validity gating on observed engagement is now standard. Metric-unit mismatch (F4): local token estimates and provider usage are close in direction but not per-turn comparable; all cost math now uses provider units, estimates only gate. In-band declarations: having the model declare work semantics in-channel costs ~+80 output and ~+66 reasoning tokens per turn (piggybacked, no turn inflation) — an affordable instrument for the calibration study, not a free one.

6. 负面结果

每个子章节都是我们原本期望提出但无法提出的主张，或者是其机制现已成为设计一部分的失败案例。分类标签（F1–F4）来自轮次级回顾，目前在测试计划中拥有黄金测试夹具。

6.1 F1 — 工作语义损失：投影在探索任务中失效

数据包 E1 (探索) A B′ C D
成本，中位数 1527 1287 1616 2445

在探索任务中，投影的成本优势消失了（1616 对 1527），且其完成率不稳定。严重失败案例（14-r2-G2-E1-C）：运行了 12 个投影轮次，生成的产物文件为零，所需的 SUBSYSTEM-MAP.md 从未生成，重读爆发集中在压缩路径上。日志中的机制清晰可见：投影保留了文件证据（路径、哈希、预览），但破坏了进度语义——什么已被映射、还剩什么、欠缺什么产物。当模型失去方向时，它会重新打开已压缩的文件（重读率这一列就是该信号），或者在最坏的情况下，原地打转进行读取和推理，而从未跨越产物生成的边界。

诚实起见，基线列也应予以考虑：A 分支也失败过一次（0/5）。在探索性任务中迷失方向在一定程度上是任务性质使然，并非纯粹是压缩病理。但配对的规律依然成立：探索所需保留的是任务状态，而提取式摘要并不携带这些状态。与此同时，分支 D 在两轮中完成了 6/6（数据集中唯一的此类单元），成本约为基线的 1.6 倍——其检查点在上下文中保留了持久的进度摘要。结论的措辞是深思熟虑的：混合+检查点在探索中换取的是完成度，而不是成本节省。

6.2 锚点的完成主张在安慰剂对照下失效

语义锚点（第 3.5 节）是针对 F1 专门构建的、无 LLM 参与的直接响应：如果悬空要素是进度状态，则在每个投影轮次中以确定性方式重申它。初步结果看起来是证实性的。然而，受控对比（带有每轮 OBS-TAIL 证据的分支 B）给出了不同的结论：

E1 分支 (n=3) 完成率 中位数轮次 中位数总 Token 数
C″ (锚点) 2/3 21 677,535
C-SB (锚点 + 旁路摘要) 1/3 21 709,726
C+PL (Token 大小匹配的安慰剂尾部) 3/3 28 882,010

一个尺寸相同但内容为空的尾部块，比包含丰富信息的锚点更可靠地完成了任务。设计日志中的裁决（R14）分离了最初解释中所混淆的变量：仪式换来的是完成，内容换来的是轨迹。 在投影轮次中追加任何相同形状的提醒，似乎都能重新引导模型走向完成；而锚点提供的可验证事实则体现在成本一栏中（C″是观察到的最便宜的 E1 分支，一次运行为 19 轮 / 538k Token）——它缩短了轨迹，减少了无序漫游，而不是提高了完成率。两项推论被列入了路线图而不是叙事中：受控的极简提示（失控的变体完成了 3/3，但成本约为 6 倍——84 轮，3.23M Token——没有任何内容的纯仪式是一种昂贵的迷信），以及一项标定研究，因为“声明为可丢弃但被重读”的现象现在可以按路径进行测量。

6.3 F2 — 逃过检查的流失：未见成效的负面探测

D1，即设计的负面区域（在激进 8k 阈值下对确切文本的依赖），未能产生预期的失败：分支 C 以低于基线的中位数成本通过了 4/4 ×3。但日志显示了它得以存活的原因，而这一原因是一个警告：压缩路径的重读率高达 .917 和 .926——模型一再请求确切内容，框架一再将其压缩掉，而重读在每次都修复了这一损失。该探测之所以能存活，是通过分期支付 T3 达成的。我们将其记录为“当前的探测手段过于薄弱”，而非“负面区域为空”；一个更强的逐字依赖探测器已在排队中（第 8 节）。普遍的教训是方法论上的：单凭验收脚本会判定这是一个干净的通过；而流失计数器才是保持其诚实的工具。

6.4 失败但被保留的微型工具

旁路摘要 (C-SB)——对工具结果进行离线 LLM 摘要——在当前的实现中违背了其自身的经济学：一次运行消耗了 186 次旁路调用 / 158k 输出 Token，最终完成度为 0/5。净流出核算（摘要成本必须低于其节省的成本）目前已成为任何重新运行的模拟级前提条件；该设计尚未被证实，但也未被证伪，暂时搁置。原生触发的非确定性 (F3)：pi 的窗口压力压缩在小数据包上仅触发了 1/3，导致两次 B′ 运行无效——基于观察到的参与度进行有效性门控现已成为标准。度量单位失配 (F4)：本地 Token 估算与服务商实际用量方向接近，但无法在逐轮层面进行比较；所有成本数学计算现在均使用服务商单位，估算仅用于门控。带内声明：让模型在通道内声明工作语义每轮增加约 80 个输出 Token 和约 66 个推理 Token（搭载传输，无轮次膨胀）——这对于标定研究来说是一个负担得起的手段，但并非免费。

7. Scope and limitations

Everything above holds within, and only within, the following frame. The measurements cover loop-internal context economics: a thin harness whose entire context assembly is observable, one provider whose cache telemetry is exposed (DeepSeek), one model generation (v4-flash), projection thresholds between 8k and 48k tokens, three task packets, n=3 per cell. Medians and paired ranks are reported; no significance claims are made or licensed at this n. Frontier agent products assemble context outside the loop (retrieval, IDE state, hidden tool output) in ways that are not observable from where we stand; nothing here measures them. Model-generation dependence is a finding, not a nuisance: which failure modes a model exhibits under compaction (and how well its self-reports calibrate) shifts with training, which is precisely why the calibration rate is proposed as a capability measurement rather than assumed constant. Extrapolation to million-token windows is a mechanism argument (§1.1 survives any window size; the discount asymmetry and T3 do not disappear), not a data claim. The small scale is a choice, not a concession: 8k–48k on a cache-observable provider is currently the only place every line of the compaction-tax ledger can be independently audited.

One further honesty note: the cost unit prices cached input at 0.1× but ignores provider-side variation in cache retention and any charge model where cached spans still occupy window; results should port across providers in shape, not in constants.

7. 范围与局限性

以上所有结论均在以下框架内成立，且仅在该框架内成立。测量涵盖了循环内部的上下文经济学：一个其整个上下文装配均可观测的微型框架，一个公开了缓存遥测的服务商（DeepSeek），一代模型（v4-flash），8k 至 48k Token 之间的投影阈值，三个任务数据包，每个单元 n=3。报告了中位数和配对排名；在此 n 值下不作任何显著性声明，亦不具备此类授权。前沿智能体产品在循环外部装配上下文（检索、IDE 状态、隐藏工具输出），其方式从我们所处的位置来看是不可观测的；这里的任何内容都无法衡量它们。模型世代依赖性是一项发现，而非干扰因素：模型在压缩下表现出何种失效模式（以及其自我报告的标定效果如何）会随着训练而改变，这正是将标定率作为能力指标来衡量、而非假定其恒定的原因。外推至百万 Token 窗口是一个机制层面的论点（第 1.1 节在任何窗口大小下均有效；折扣不对称性和 T3 并未消失），而非数据层面的断言。选择小规模是一个决定，而非妥协：在支持缓存观测的服务商上进行 8k–48k 的测试，是目前唯一可以对压缩税账本的每一行进行独立审计的地方。

另一个诚实性说明是，成本单位将缓存输入按 0.1 倍计费，但忽略了服务商端在缓存保留期方面的差异，以及缓存区间仍占用窗口的任何计费模型；实验结果在形态上应当跨服务商迁移，但在常数上则不然。

8. Roadmap

Every open item carries its gate; none is a promise.

  1. SWEEP-R2 — threshold curve: R1-C at 4k/16k/64k and D1-C at 4k, six runs, same audit. Gate for: the sweet-spot boundary, and for upgrading §5.1 from a point to a curve.
  2. Stronger verbatim probe — a task where re-reading cannot repair exactness loss (or is made expensive). Gate for: any claim about the direct-transform row of the dispatch policy.
  3. Second trajectory×quality data point — required before the §5.2 proxy hypothesis is used anywhere.
  4. Governed nudge (NUDGE-GOV) — the placebo result implies a minimal ritual tail; ungoverned it cost 6×. Injection cap plus stop-on-empty-pending. Not scheduled until Branch-B follow-ups clear.
  5. Calibration study (C‴) — measure "declared disposable but re-read" as a per-model calibration rate; the declaration tax numbers (§6.4) are in hand. This is the paper-axis item: the rate is a capability contour, usable across model generations.
  6. Ledger continuation (WS-6) — restore work-semantic state across sessions from the persisted ledger. Gated on dogfooding evidence and retention rules.
  7. Provider-compaction arm (E) — when a cache-observable provider exposes a server-side compaction API, compare it as an arm rather than rewriting the thesis around it.
  8. Dispatch policy hardening — the R2 mapping (refactor → projection; exploration/completion-first → checkpoint; exploration/cost-first → native or baseline; direct-transform → no special handling at current probe strength) ships as a recommendation layer only, until it survives held-out task families.

The stop conditions are as much a part of the roadmap as the queue: a mechanism that lowers tokens but raises artifact failure stops; a mechanism that wins under only one prompt shape stays experimental; a provider feature that obsoletes a local rule retires the rule and keeps the measurement harness.

8. 路线图

每一个开放项都带有其门控条件，均非承诺。

  1. SWEEP-R2 — 阈值曲线：在 4k/16k/64k 下的 R1-C 以及在 4k 下的 D1-C，共六次运行，采用相同的审计。门控条件：寻找甜点区边界，并将第 5.1 节从一个点升级为一条曲线。
  2. 更强的逐字依赖探测 — 一项重读无法修复确切性损失（或使重读变得昂贵）的任务。门控条件：对调度策略的直接转换行提出任何主张。
  3. 第二个“轨迹 × 质量”数据点 — 在第 5.2 节的代理假设被用于任何地方之前，这是必须具备的前提。
  4. 受控微调 (NUDGE-GOV) — 安慰剂结果意味着需要一个极简的仪式性尾部；在未受控状态下，其成本达到了 6 倍。注入上限加上空悬挂停止机制。在分支 B 的后续跟进清理完毕前不予安排。
  5. 标定研究 (C‴) — 将“声明为可丢弃但被重读”作为每个模型的标定率进行测量；声明税数据（第 6.4 节）已掌握。这是论文轴线项：该比率是一个能力轮廓，可跨模型世代使用。
  6. 账本延续 (WS-6) — 从持久化账本中跨会话恢复工作语义状态。门控条件：狗食验证证据与保留规则。
  7. 服务商压缩分支 (E) — 当支持缓存观测的服务商公开服务器端压缩 API 时，将其作为分支进行对比，而不是围绕它重写论文。
  8. 调度策略硬化 — R2 映射（重构 → 投影；探索/完成度优先 → 检查点；探索/成本优先 → 原生或基线；直接转换 → 在当前探测强度下无特殊处理）仅作为推荐层发布，直至其在保留的任务家族中存活下来。

停止条件与队列一样是路线图的一部分：降低 Token 但提高产物失效的机制必须停止；仅在一种提示形状下胜出的机制保持实验性；使本地规则废弃的服务商特性将使该规则退役，并保留测量框架。

9. A note on how this was built

The repository's code was written by LLM coding agents working from self-contained task packets (specification, exact file list, acceptance criteria, prohibited zones), test-first, with the packets and their revision history preserved in the repo's goal ledger. Human involvement was deliberately concentrated at the judgment layer: parameter choices, quality review, gate rulings, and the decision of what may be claimed — none of which was delegated. Two operational incidents during the project (a summary-induced context loss affecting the supervising session itself, and a hard-zone violation by a weaker model) were captured, rolled back, and codified into the collaboration protocol rather than discarded. We note this both for provenance and because it is a small instance of the paper's subject: the workflow was itself an exercise in deciding which context to trust, which to verify, and which to deliberately forget.
